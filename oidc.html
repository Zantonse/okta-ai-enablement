Section 01

What OIDC Is and Why It Exists

The Core Problem OAuth Left Unsolved

OAuth 2.0 is an authorization framework. It answers: "What is this app allowed to do?" It issues access tokens that let a client act on behalf of a user — but says nothing about who that user is.

Before OIDC, every major provider (Google, Facebook, Twitter) invented its own non-interoperable layer on top of OAuth to express identity. No standard claims, no standard token format, no standard discovery mechanism.

OAuth 2.0

Authorization — what can the app do?

Issues access tokens for resource access. Does not identify the user. Opaque to the client about who logged in.

OpenID Connect (OIDC)

Authentication — who is the user?

Adds an identity layer on top of OAuth 2.0. Standardized claims, ID Token, and discovery. Published Feb 26, 2014 by the OpenID Foundation.

The OIDC Formula

What OIDC Adds to OAuth 2.0
OIDC = OAuth 2.1 (authorization) + ID Token + UserInfo endpoint + Discovery + Session management

Specifically, OIDC introduces:

  • A new ID Token (always a JWT) asserting authentication facts about the user
  • A mandatory openid scope that signals an identity request
  • A standard set of claims (sub, name, email, picture, etc.)
  • A UserInfo endpoint for retrieving additional user attributes
  • Discovery via /.well-known/openid-configuration
  • Session management and logout specifications
  • Dynamic Client Registration

Key Vocabulary

Term Meaning
OpenID Provider (OP) The identity provider — authenticates users and issues tokens. Okta acts as an OP. Also called IdP or Authorization Server.
Relying Party (RP) The client application that outsources authentication to the OP. Your web app is the RP.
End-User The human being being authenticated.
ID Token JWT issued by the OP asserting who the user is. Always a JWT — never opaque.
UserInfo Endpoint OP endpoint where the RP retrieves additional user claims using an access token.
Claims Key-value pairs asserting facts about the user or authentication event (e.g., sub, name, email, acr).
Why OIDC vs Rolling Your Own on OAuth

Interoperability: standard claims and discovery mean any OIDC client works with any OIDC OP. Security built in: nonce prevents replay attacks, signature verification prevents tampering. No password storage at the RP. Extensible: ACR, AMR, CIBA, DCR all build on core OIDC.

Section 02

Core Concepts

The ID Token

The ID Token is OIDC's central artifact. It is always a JWT (JSON Web Token, RFC 7519) — never an opaque token. It represents proof of the authentication event.

⚠
What the ID Token is NOT for

Calling APIs. The ID token is a statement about the user for the client. It should never be sent to a resource server as an authorization credential. That is what the access token is for.

Structure: three Base64url-encoded parts separated by dots — Header.Payload.Signature

Decoded Header — Okta
{
  "alg": "RS256",   // signature algorithm — never trust "none"
  "kid": "abc123"   // key ID — use to look up the signing key in JWKS
}
Decoded Payload — Full Okta Example
{
  "sub":                "00u47ijy7sRLaeSdC0g7",       // required: subject
  "name":               "Jane Doe",
  "email":              "jane@example.com",
  "preferred_username": "jane.doe@example.com",
  "given_name":         "Jane",
  "family_name":        "Doe",
  "picture":            "https://example.com/photo.jpg",
  "locale":             "en-US",
  "zoneinfo":           "America/Los_Angeles",
  "iss":                "https://dev-123456.oktapreview.com/oauth2/default",  // required
  "aud":                "0oa48e74ox4t7mQJX0g7",      // required: your client_id
  "iat":                1661289624,                  // required: issued at
  "exp":                1661293224,                  // required: expiration
  "nonce":              "n-0S6_WzA2Mj",              // replay protection
  "auth_time":          1661289603,                  // when user actually authed
  "acr":                "urn:okta:loa:1fa:any",      // auth context class
  "amr":                ["pwd"],                      // auth methods used
  "azp":                "0oa48e74ox4t7mQJX0g7",      // authorized party
  "jti":                "ID.dz6ibX-YnBNlt14huAtBULam_Z0",
  "at_hash":            "w6BLQV3642TKWvaVwTAJuw",    // access token hash
  "ver":                1
}

Required ID Token Claims

These MUST always be present in every ID token. Validation fails if any are missing.

Claim Type Description
iss REQUIRED string Issuer — the OP's URL. Must be https://. Must exactly match the issuer in the discovery document.
sub REQUIRED string Subject — unique identifier for the user at this OP. Max 255 ASCII chars. Persistent and non-reassignable.
aud REQUIRED string or array Audience — must contain the client's client_id. If an array, also validate against azp.
exp REQUIRED number Expiration — Unix timestamp. Token MUST be rejected if now > exp.
iat REQUIRED number Issued At — Unix timestamp of when the token was created.

Optional ID Token Claims

Claim Description
nonce CONDITIONAL Echo of the nonce sent in the authorization request. MUST be present if nonce was sent. Prevents replay attacks.
auth_time Unix timestamp of when the user last authenticated with credentials (not SSO — actual credential entry).
acr Authentication Context Class Reference — the assurance level achieved (e.g., urn:okta:loa:2fa:any).
amr Authentication Methods References — array of what methods were used (e.g., ["pwd", "mfa", "hwk"]).
azp Authorized Party — the client_id of the client the token was issued to. Used when aud is an array.
at_hash Access token hash — first half of SHA-256 of access_token, base64url encoded. Binds the access token to this ID token.
c_hash Code hash — same principle as at_hash but for the authorization code. Used in hybrid flow.
jti JWT ID — a unique identifier for this specific token instance. Enables per-token replay detection.

The openid Scope — The Critical Trigger

Adding scope=openid to any OAuth 2.0 authorization request transforms it from a pure authorization request into an OIDC authentication request.

Dimension Without openid With openid
Protocol Pure OAuth 2.0 authorization OIDC authentication
Token response Only access_token access_token + id_token
User identity No claims about who the user is ID token claims assert user identity
Nonce Not applicable Nonce MUST be validated if sent in request
UserInfo endpoint Not meaningful Provides user profile claims
The Scope Rule

openid is required. You cannot do OIDC without it. It is the switch that causes the OP to issue an ID token and activates all OIDC spec requirements.

Standard OIDC Scopes

Scope Claims Returned Notes
openid sub Required. Without this, it is not OIDC.
profile name, given_name, family_name, middle_name, nickname, preferred_username, profile, picture, website, gender, birthdate, zoneinfo, locale, updated_at Full user profile claims.
email email, email_verified email_verified is a boolean.
address address (JSON object) Contains street_address, locality, region, postal_code, country.
phone phone_number, phone_number_verified phone_number_verified is a boolean.

Nonce — Replay Attack Prevention

The nonce is a cryptographically random value the RP generates and sends in the authorization request. The OP echoes it back in the ID token. The RP must store the nonce before the redirect and verify the match after receiving the token.

Nonce in Authorization Request
/authorize?...&nonce=n-0S6_WzA2Mj

// After receiving ID token, RP must verify:
assert(id_token.nonce === stored_nonce)
⚠
Replay Attack Risk

Without nonce validation, an attacker who captures a valid ID token can replay it to authenticate as that user. Always generate a fresh nonce per request, store it (e.g., in session), and verify it after token receipt.

Subject Identifier Types

Public (default)

The same sub value is returned for the same user to all RPs. Simple to implement, but allows multiple RPs to correlate users by sub.

Okta default. Most common in enterprise use.

Pairwise

A different sub is generated per (RP, user) pair. RP-A and RP-B cannot correlate users even if they share the same user base.

sub = SHA-256(sector_identifier + user_id + salt)

Trade-off: More complex, harder to debug, cross-app linking is impossible.

Section 03

OIDC Flows

Authorization Code Flow with PKCE (Recommended for Everything)

This is the correct modern flow for web apps, SPAs, and native apps. When scope=openid is included, the token endpoint returns an ID token alongside the access token.

Authorization Code + PKCE + OIDC — Full Request Sequence
1

RP Generates PKCE Verifier and Challenge

Before any redirect, the RP creates a random code verifier and its SHA-256 hash. These are stored locally — the verifier is never sent until step 4.

PKCE Setup
// Generate a 32-byte random value, base64url-encode it
code_verifier  = base64url(random_bytes(32))   // e.g., "abc123..."
code_challenge = base64url(SHA-256(code_verifier))
// Store code_verifier in session — send only code_challenge
2

Authorization Request — Browser Redirect to OP

RP redirects the user's browser to the OP's authorization_endpoint. The nonce and PKCE challenge go here.

HTTP Request
GET /oauth2/default/v1/authorize
  ?client_id=0oa48e74ox4t7mQJX0g7
  &response_type=code
  &scope=openid profile email
  &redirect_uri=https://app.example.com/callback
  &state=zCz0nXe5DP9lkH0oBPVB
  &nonce=n-0S6_WzA2Mj
  &code_challenge=E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM
  &code_challenge_method=S256
3

User Authenticates — OP Returns Authorization Code

After the user authenticates at the OP, a short-lived authorization code is returned via redirect. The code is single-use and has a short TTL (typically 60–300 seconds). The state parameter is echoed back for CSRF protection.

HTTP Response
HTTP/1.1 302 Found
Location: https://app.example.com/callback
  ?code=SplxlOBeZQQYbYS6WxSbIA
  &state=zCz0nXe5DP9lkH0oBPVB  // must match what was sent
4

Token Exchange — Back-Channel (Server to Server)

The RP exchanges the code for tokens. This happens server-to-server — no browser involvement. The code_verifier (never sent before) proves the same party that started the flow is completing it.

HTTP Request
POST /oauth2/default/v1/token HTTP/1.1
Host: dev-123456.okta.com
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=SplxlOBeZQQYbYS6WxSbIA
&redirect_uri=https://app.example.com/callback
&client_id=0oa48e74ox4t7mQJX0g7
&code_verifier=abc123...   // the original verifier — OP will SHA-256 and compare
5

Token Response — ID Token + Access Token Returned

The OP returns tokens. The id_token is for the RP to validate and use for authentication. The access_token is for calling APIs. They serve different purposes and go to different places.

JSON Response
{
  "token_type":    "Bearer",
  "expires_in":   3600,
  "access_token": "eyJraW...vSqrH1Q",    // for APIs
  "id_token":     "eyJraW...WlB8Y9pQ",   // for the RP — validate this
  "refresh_token":"hC9XDE5BbUfEBT8",
  "scope":        "openid profile email"
}

Implicit Flow (Deprecated)

⚠
Deprecated in OAuth 2.1

In the implicit flow (response_type=token id_token), tokens are returned directly in the URL fragment from the authorization endpoint. This causes token leakage in browser history and referrer headers. Do not recommend to customers. The OIDC Core spec retains it for backwards compatibility only.

Implicit Flow response_type Values
response_type=id_token         // Returns only ID token from auth endpoint
response_type=token id_token   // Returns access token + ID token from auth endpoint
// Both appear in the URL fragment (#), not query string

Hybrid Flow

The hybrid flow returns some tokens from the authorization endpoint and others from the token endpoint. Use when the RP needs immediate proof of authentication (front-channel ID token) but also needs an access token via the more secure back-channel.

response_type From Authorization Endpoint From Token Endpoint
code id_token code + ID token access token
code token code + access token ID token
code id_token token code + ID token + access token —
c_hash Binding in Hybrid Flow

The front-channel ID token in hybrid flow contains c_hash — a hash of the authorization code. The RP must verify c_hash to bind the tokens together and confirm they came from the same flow.

Section 04

Discovery

The /.well-known/openid-configuration Document

Every OIDC provider MUST publish a JSON metadata document at {issuer}/.well-known/openid-configuration. This is defined in OpenID Connect Discovery 1.0, extended by RFC 8414. The /.well-known/ convention comes from RFC 8615 — a reserved URI space for standardized host metadata.

{

  "authorization_endpoint":               "https://dev-123456.okta.com/oauth2/default/v1/authorize",

Required Discovery Fields

Field Purpose
issuer Canonical issuer URL. MUST exactly match iss in all tokens issued by this server.
authorization_endpoint Where users are redirected to authenticate. Entry point for all interactive flows.
token_endpoint Where authorization codes are exchanged for tokens. Back-channel, server-to-server.

end_session_endpoint RP-initiated logout endpoint. Redirect here to terminate the OP session.
introspection_endpoint Token introspection (RFC 7662). Lets resource servers validate opaque tokens.
revocation_endpoint Token revocation (RFC 7009). Invalidate access or refresh tokens.
code_challenge_methods_supported PKCE methods. Always require S256 for new clients — never accept plain.
id_token_signing_alg_values_supported Must include RS256. Never trust none.

Org AS vs Custom AS — Capability Comparison

Capability Org AS Custom AS
SSO with OIDC Yes Yes
Custom scopes No Yes
Custom claims No Yes
Access policies No Yes
Protecting your own APIs No Yes
Mint Okta API tokens Yes No
Token validation by your app Not recommended Yes
Practical SE Rule

If a customer is building a custom API or wants custom claims in tokens, they need a Custom AS. For basic SSO with standard profile claims, the Org AS is sufficient. When in doubt, default to the Custom AS — it is a superset of Org AS capabilities.

Section 05

ID Token Validation

These are the steps an RP MUST perform after receiving an ID token. Complete all steps. Never use a token before validation is finished.

  1. Decode the Header — Check Algorithm

    Base64url-decode the JWT header (do not verify yet). Extract kid and alg. Confirm alg is on your allowlist. If alg is none, reject immediately.

    Decoded Header
    {
  "kid": "abc123",   // use this to find the signing key in JWKS
  "alg": "RS256"    // must be an expected algorithm — NEVER "none"
}
  2. Fetch JWKS and Match the Key

    Retrieve the JWKS from the jwks_uri in the discovery document. Find the key where jwk.kid === header.kid. Cache by kid; re-fetch on unknown kid (supports key rotation).

    JWKS Response — RSA Public Key
    GET {jwks_uri}   // e.g., https://dev-123456.okta.com/oauth2/default/v1/keys

{
  "keys": [{
    "kty": "RSA",
    "kid": "abc123",   // match this to header.kid
    "use": "sig",
    "alg": "RS256",
    "n":   "sI6M...base64url...",
    "e":   "AQAB"
  }]
}
  3. Verify the Signature

    Using the RSA public key from the JWKS, verify that the JWT signature is valid over the concatenation base64url(header) + "." + base64url(payload). If verification fails, reject — the token was tampered with or forged.

  4. Validate Claims

    Verify each required claim against strict rules. Any mismatch is a hard rejection.

    Claim Validation Rule
    iss MUST exactly match the issuer from the discovery document. Including any trailing slash. String equality — no normalization.
    aud MUST contain your client_id. If aud is an array and has other entries, verify azp equals your client_id.
    exp MUST be in the future: now < exp. Allow a small clock skew (e.g., ±60 seconds) to account for time drift between systems.
    iat SHOULD be recent. Guard against forwarded or reused tokens by rejecting tokens with a very old iat.
    nonce If a nonce was sent in the authorization request, this MUST match the stored nonce exactly. Missing nonce when one was sent is a hard failure.
    acr If you specified acr_values in the request, verify the returned value meets or exceeds your required assurance level.
  5. Validate at_hash if Access Token is Present

    When an access token is returned alongside the ID token and the ID token contains at_hash, verify the binding. This prevents an attacker from substituting a different access token.

    at_hash Binding Formula
    // 1. Take the raw bytes of the access_token ASCII string
// 2. Compute SHA-256
// 3. Take the first half (left 128 bits / 16 bytes)
// 4. Base64url-encode that half

expected_at_hash = base64url( SHA-256(access_token_bytes)[0:16] )

assert id_token.at_hash === expected_at_hash
// mismatch = access token was swapped — reject both tokens

Common Mistakes

Mistake 1 — Sending the ID Token to an API
WRONG
Authorization: Bearer <id_token>   // Never do this

The ID token is for the RP only. APIs do not know how to validate it as authorization. It contains PII that should not be broadcast to every service. Send the access_token to APIs.

Mistake 2 — Using the Access Token to Identify the User in the RP

The access token is for resource servers — its format and contents are opaque to the RP and can change without notice. Always use id_token.sub as the user identifier in the RP.

Mistake 3 — Skipping Nonce Validation

Skipping nonce validation opens the door to replay attacks where stolen ID tokens are reused against the same or a different RP. If you sent a nonce, validating it is not optional.

Mistake 4 — Trusting alg: none

Some early JWT libraries would skip signature verification if alg was "none". Always whitelist the expected algorithms (e.g., ["RS256"]) before processing. Never use a library that defaults to trusting any algorithm header.

Mistake 5 — Trusting Unverified Claims Before Signature Check

Never read iss from the payload to decide which key to use before verifying the signature. The header kid is the lookup key. Verify first, read payload claims after.

Mistake 6 — Not Pinning the iss in Multi-Tenant Apps

In multi-tenant applications that accept tokens from multiple issuers, each token must be validated against the specific expected issuer for that tenant. A token from one tenant's OP must never be accepted by another tenant's resource.

Validation Checklist — Quick Reference

1. Decode header — verify alg is expected, never "none"
2. Fetch JWKS — match key by kid
3. Verify signature using RSA public key
4. Validate claims: iss exact match, aud contains client_id, exp > now, nonce matches stored value
5. Validate at_hash binding if access token is present

Section 6

Session Management & Logout

OIDC defines four complementary specifications for session lifecycle management. Each addresses a different failure mode — from the simplest user-initiated logout to server-to-server signals that work even when the browser is closed.

6.1 — RP-Initiated Logout

The client initiates the logout sequence by redirecting the user's browser to the OP's end_session_endpoint. This is the most common logout pattern.

HTTP 
# Step 1 — RP clears its local session (invalidate cookies, clear tokens)
# Step 2 — RP redirects the browser:

GET https://dev-123456.okta.com/oauth2/default/v1/logout
  ?id_token_hint=eyJraW...WlB8Y9pQ
  &post_logout_redirect_uri=https://app.example.com/logged-out
  &state=abc123

# Parameters:
#   id_token_hint             — Previously issued ID token (RECOMMENDED).
#                               Identifies the user and the client without
#                               requiring a separate auth check.
#   post_logout_redirect_uri  — Where to send the user after logout.
#                               Must be pre-registered on the app.
#   state                     — Opaque value passed back unchanged in redirect.
1
RP clears local session Invalidates its own session cookie and discards stored tokens before redirecting.
2
Browser redirected to end_session_endpoint RP sends the user's browser to Okta's logout URL with id_token_hint and redirect URI.
3
OP invalidates the SSO session Okta destroys its central session cookie scoped to the Okta domain.
4
OP triggers logout at connected RPs Okta notifies other relying parties that have an active session — via front-channel iframes or back-channel HTTP POSTs.
5
OP redirects to post_logout_redirect_uri User lands on the app's post-logout page with the state parameter echoed back.

6.2 — Front-Channel Logout (iframe-based)

When a user's session ends at the OP, the OP renders a page containing hidden iframes — one per connected RP. Each iframe loads the RP's registered frontchannel_logout_uri, causing the RP's server to receive a GET request and invalidate its session.

How it works

Each RP registers a frontchannel_logout_uri. When logout is triggered, the OP inserts iframes:

HTTP 
# RP registers:
frontchannel_logout_uri = https://app.example.com/oidc-logout

# OP calls via hidden iframe (browser-rendered):
GET https://app.example.com/oidc-logout
  ?iss=https://dev-123456.okta.com/oauth2/default
  &sid=session123
Third-party cookie blocking in modern browsers (Safari ITP, Chrome Phase-Out) can silently break front-channel logout. When the OP's logout page loads iframes pointing to RP domains, the RP's session cookies may not be sent if classified as third-party. This makes front-channel logout unreliable as a standalone mechanism — always configure back-channel logout as the primary method.

6.3 — Back-Channel Logout

The OP makes a direct server-to-server POST to each RP's registered backchannel_logout_uri. The request body contains a Logout Token — a signed JWT specifically designed for logout notifications.

Logout Token (JWT Payload) 
{
  "iss": "https://dev-123456.okta.com/oauth2/default",
  "sub": "00u47ijy7sRLaeSdC0g7",
  "aud": "0oa48e74ox4t7mQJX0g7",
  "iat": 1661289624,
  "jti": "unique-logout-token-id",
  "events": {
    "http://schemas.openid.net/event/backchannel-logout": {}
  },
  "sid": "session-id"
}
HTTP POST (OP → RP) 
POST /oidc-backchannel-logout HTTP/1.1
Host: app.example.com
Content-Type: application/x-www-form-urlencoded

logout_token=eyJhbGciOiJSUzI1NiIsImtpZCI6ImFiYzEyMyJ9...

# RP validates the logout token (same steps as ID token validation),
# looks up the session matching "sid" or "sub", and invalidates it.
# RP responds 200 OK on success, 400 on validation failure.
Back-channel logout does not depend on the user's browser being open. The OP calls your server directly. It is immune to third-party cookie blocking and continues to work when users close tabs or their browser crashes.

6.4 — Session Management: check_session_iframe Polling

The OP publishes a check_session_iframe URL in the discovery document. The RP embeds this URL in a hidden iframe and uses postMessage to periodically poll whether the OP session is still active — without making a full network round-trip.

JavaScript 
// RP embeds the OP's check_session_iframe URL as a hidden iframe
// Then polls every 2 seconds via postMessage:
const opIframe = document.getElementById('op-iframe');

setInterval(() => {
  opIframe.contentWindow.postMessage(
    client_id + ' ' + session_state,
    op_iframe_origin
  );
}, 2000);

// OP iframe responds with one of three values:
window.addEventListener('message', (e) => {
  if (e.origin !== op_iframe_origin) return;

  if (e.data === 'unchanged') {
    // Session still active — do nothing
  } else if (e.data === 'changed') {
    // Session changed — trigger silent re-authentication
    doSilentReauth(); // /authorize?prompt=none
  } else if (e.data === 'error') {
    // Cannot determine state — may indicate session ended
    handleSessionError();
  }
});

Silent Re-authentication on "changed"

When the iframe returns changed, the RP initiates a silent re-authentication request with prompt=none. If the user is still logged in, Okta issues new tokens silently. If not, Okta returns login_required and the RP redirects the user to a full login page.

6.5 — Okta Universal Logout (ULO)

Universal Logout is an Okta-specific security feature that goes beyond standard OIDC session termination. When Okta detects a security event — a compromised account signal, admin-forced logout, or a Shared Signals Framework (SSF) event — it can proactively push logout signals to all connected applications that support back-channel logout.

Universal Logout (ULO) extends back-channel logout into a security response tool. Unlike user-initiated logout, ULO is triggered by Okta's threat detection or by admin action. Applications that have registered a backchannel_logout_uri receive the logout token automatically — no user action or browser session required. This is critical for responding to credential compromise: you can instantly revoke all active sessions across every connected app in seconds.

Section 7

Advanced OIDC

7.1 — CIBA: Client-Initiated Backchannel Authentication

CIBA (OpenID Connect Client-Initiated Backchannel Authentication Core 1.0) extends OIDC for decoupled, out-of-band authentication. There are no browser redirects. The device initiating the request and the device where the user authenticates are different.

Problem CIBA Solves

A consumption device (bank POS terminal, call center agent's screen, hospital workstation) needs to authenticate a user who is physically present but whose primary authentication device (their smartphone) is separate. Standard redirect flows require a browser — CIBA does not.

CIBA Flow — 5 Steps

1
Consumption Device sends Backchannel Auth Request Client POSTs directly to the OP's Backchannel Authentication Endpoint — no browser redirect. Includes a hint identifying the target user (login_hint, id_token_hint, or login_hint_token) and a human-readable binding_message shown on both devices.
2
OP returns auth_req_id OP validates the request and returns a transaction identifier (auth_req_id) plus expires_in and interval for polling. No tokens yet.
3
OP contacts the user's Authentication Device OP sends a push notification, SMS, or in-app prompt to the user's registered authentication device. The binding_message is displayed on both the consumption device and the auth device so the user can verify they match.
4
User authenticates on their device The user approves (or denies) the request on their authentication device. The OP records the completed authentication event against the auth_req_id.
5
Tokens delivered to consumption device Depending on the delivery mode (Poll, Ping, or Push), the consumption device receives the ID token and access token. Grant type: urn:openid:params:grant-type:ciba.
HTTP — CIBA Request 
POST /oauth2/default/v1/bc-authorize HTTP/1.1
Host: dev-123456.okta.com
Content-Type: application/x-www-form-urlencoded
Authorization: Basic <client_credentials>

scope=openid profile
&login_hint=jane@example.com
&binding_message=Authorize your bank transfer of $500
&requested_expiry=120

--- Response ---
{
  "auth_req_id": "1c266114-a1be-4252-8ad1-04986c5b9ac1",
  "expires_in": 120,
  "interval": 5
}

Three Token Delivery Modes

Poll

Client polls the token endpoint with auth_req_id every interval seconds until authentication completes or the request expires. Simplest to implement.

Ping

Client registers a notification endpoint. OP POSTs a notification when the user authenticates. Client then fetches tokens in a separate request. Reduces unnecessary polling.

Push

OP directly POSTs all tokens to the client's registered notification endpoint as soon as authentication completes. Lowest latency, requires a publicly accessible endpoint.

Okta supports CIBA on Identity Engine (IEng) orgs. Real-world use cases: banking PSD2 strong customer authentication (SCA), call center agent identity verification, healthcare patient consent workflows, and POS terminal payment authorization. Auth0 does not support CIBA as of 2026.

7.2 — ACR Values: Authentication Context Class Reference

The acr_values request parameter tells the OP what level of authentication assurance is required. The OP returns the achieved level in the acr claim of the ID token and access token. Resource servers can then enforce access control based on the ACR.

Okta Predefined ACR Values

ACR Value Meaning Authenticators That Satisfy It
urn:okta:loa:1fa:any Any single factor Any single authenticator
urn:okta:loa:1fa:pwd Password specifically Password only
urn:okta:loa:2fa:any Any two factors Any combination of two authenticators
urn:okta:loa:2fa:any:ifpossible Two factors if enrolled, otherwise one Graceful MFA — does not fail if MFA not enrolled
phr Phishing-Resistant FIDO2 / WebAuthn only (origin-bound)
phrh Phishing-Resistant + Hardware-Protected Okta Verify with TPM (Windows) or Secure Enclave (iOS/macOS)

Step-Up Authentication Pattern

1
User logs in normally Standard login with any single factor. ID token contains acr=urn:okta:loa:1fa:any.
2
User requests a sensitive resource e.g., View social security number, initiate a wire transfer, or access an admin console.
3
App checks the acr claim Current token only satisfies 1FA. The resource requires 2FA (or phr). Step-up is needed.
4
App redirects to /authorize with elevated acr_values acr_values=urn:okta:loa:2fa:any&max_age=0 — max_age=0 forces re-authentication regardless of existing session.
5
Okta prompts for the required additional factor User completes MFA. New tokens are issued with acr=urn:okta:loa:2fa:any.
6
Resource server validates the acr claim Grants access only when the token's acr meets the required level. Rejects tokens below the threshold.
If the user cannot satisfy the required ACR level (e.g., not enrolled in FIDO2 but acr_values=phr was requested), Okta returns error=unmet_authentication_requirements. Your app must handle this gracefully — typically by routing the user to an enrollment flow.

7.3 — AMR Values: Authentication Methods References (RFC 8176)

While ACR describes the class of authentication achieved, AMR describes the specific methods that were used. The amr claim is an array of strings in the ID token. Resource servers can use AMR to make fine-grained access decisions.

AMR Value Authentication Method Notes
pwd Password Knowledge-based, phishable
otp One-time password (TOTP/HOTP) e.g., Google Authenticator, Authy
sms SMS one-time code Phishable; avoid for high-assurance
mfa Multiple factors used Present alongside other AMR values
hwk Hardware-secured key Key stored in TPM or Secure Enclave
kba Knowledge-based authentication Security questions
fido FIDO authenticator WebAuthn / FIDO2 — phishing-resistant
pop Proof-of-possession key Demonstrates key ownership

Example: Phishing-Resistant MFA (Okta Verify on Secure Enclave)

ID Token Claims 
{
  "amr": ["mfa", "hwk", "fido"],
  "acr": "phrh"
}

Read as: "Multiple factors were used (mfa). The key is hardware-protected (hwk) and FIDO-bound (fido). The achieved ACR is phishing-resistant + hardware-protected (phrh)."

7.4 — Claims Request Parameter

The claims request parameter gives the RP precise control over which claims are returned and from which endpoint (ID token vs. UserInfo). Claims marked essential: true must be returned by the OP or the request fails.

Authorization Request — claims parameter (URL-encoded JSON) 
// The "claims" parameter value (decoded for readability):
{
  "id_token": {
    "email":  { "essential": true },
    "acr":   { "essential": true, "values": ["phr"] }
  },
  "userinfo": {
    "given_name":  null,
    "family_name": null,
    "address":    { "essential": true }
  }
}

// In the authorization request URL:
// &claims=%7B%22id_token%22%3A%7B%22email%22%3A%7B%22essential%22%3Atrue%7D...
Claims with null (no options object) are requested but not essential — the OP should return them if available. Claims with "essential": true must be returned. If a required ACR value cannot be satisfied, the OP returns error=unmet_authentication_requirements rather than issuing a downgraded token.

OIDC vs. SAML

8.1 — Full Protocol Comparison

Dimension SAML 2.0 OIDC 1.0
Published OASIS SAML 2.0 (2005) OpenID Connect Core 1.0 (February 2014)
Data format XML JSON / JWT
Base protocol Standalone specification Built on top of OAuth 2.0
Token type XML Assertion JWT (ID Token)
Transport HTTP redirects + HTML form POST HTTP redirects + REST API
Mobile support Poor — browser-centric XML POST forms Excellent — PKCE flows for native apps
API support Poor — REST APIs expect JWTs Excellent — access token for every API call
Key management X.509 certificates (manual rotation, outage risk) JWKS auto-rotation — no cert coordination
Discovery SAML metadata XML (manual exchange) /.well-known/openid-configuration JSON (automated)
Encryption XML Encryption JWE (optional) or TLS
Session logout SAML SLO — fragile, prone to failures Multiple mechanisms: RP-initiated, front-channel, back-channel
Developer experience Complex — XML signature parsing, sparse modern libraries Simple — JSON, well-supported SDK ecosystem
AI / agent use Impractical — XML parsing, no token refresh Native — JSON-native, JWT, PKCE, CIBA for human-in-loop

8.2 — Where Each Protocol Belongs

Where SAML Still Makes Sense

  • Legacy enterprise SaaS with long-established SAML integrations (Salesforce Classic, SAP, older HR platforms)
  • Corporate VPN and desktop SSO environments baked into existing IdP contracts
  • Browser-only B2E apps with infrequent logins where session management simplicity matters more than token flexibility
  • Governance and compliance workflows already built around SAML audit trails
  • Integrations that have been running without incident for years — "if it ain't broke, don't fix it"

Where SAML Actively Fails

  • Mobile apps — SAML's XML POST redirect flow does not function in native iOS/Android apps
  • REST APIs — APIs expect JWTs; SAML assertions require significant adaptation
  • Token refresh — SAML has no native access token refresh mechanism
  • Certificate rotation — X.509 cert expiry is a leading cause of identity-related production outages
  • Modern dev patterns — Step-up auth, incremental consent, PKCE, CIBA are all awkward or impossible in SAML
  • AI / agent flows — Machine clients cannot participate in browser-redirect XML POST flows

8.3 — Migration: Dual-Stack is Realistic

Pattern 1 — Side-by-Side (Dual-Stack)

Run SAML and OIDC simultaneously on the same Okta org. Migrate application by application at your own pace. Centrally enforce consistent policies across both protocols from a single Okta policy engine.

Pattern 2 — Protocol Bridge

Use Okta as a bridge: inbound SAML assertions from a legacy IdP, outbound OIDC tokens to modern apps. Legacy systems keep sending SAML; modern apps get JWTs. Zero disruption to existing integrations.

Pattern 3 — Progressive Replacement

Default to OIDC for every new app or major rewrite. Do not migrate working SAML integrations unless they are causing active pain (cert rotation incidents, mobile requirement, API access need).

Migration risks to document before starting: (1) sub (OIDC) vs. NameID (SAML) — ensure consistent user identity mapping across the cutover. (2) Group attribute format differences — SAML multi-value attributes vs. OIDC groups claim array. (3) Session lifetime expectations — SAML session duration behavior may differ from OIDC token lifetimes. (4) Audit all existing SAML certificate expiry dates before beginning migration work.

9.3 — HTTP Cheat Sheet: Key Okta OIDC Endpoints

All commands use dev-123456.okta.com with the default custom authorization server as the base. Substitute your Okta domain and AS ID.

1. Fetch the Discovery Document
curl 
curl -s https://dev-123456.okta.com/oauth2/default/.well-known/openid-configuration | jq .

# Returns the full metadata document: all endpoint URLs, supported algorithms,
# scopes, grant types, and JWKS URI. The canonical starting point for any integration.
2. Fetch JWKS for Signature Verification
curl 
curl -s $(curl -s https://dev-123456.okta.com/oauth2/default/.well-known/openid-configuration \
  | jq -r .jwks_uri) | jq .

# Returns the public key set. Match the "kid" in a JWT header to the
# key in this response to verify the token's signature.
# Okta rotates keys periodically — always fetch from jwks_uri, never hardcode.
3. Call the UserInfo Endpoint
curl 
curl https://dev-123456.okta.com/oauth2/default/v1/userinfo \
  -H "Authorization: Bearer <access_token>"

# Returns JSON object with claims matching the scopes granted.
# The "sub" in the response MUST match the "sub" in the ID token.
# Use for richer profiles without bloating the ID token.
4. RP-Initiated Logout
curl 
curl -G "https://dev-123456.okta.com/oauth2/default/v1/logout" \
  --data-urlencode "id_token_hint=<id_token>" \
  --data-urlencode "post_logout_redirect_uri=https://app.example.com/logout" \
  --data-urlencode "state=abc123"

# Initiates RP-initiated logout. In production this is a browser redirect, not
# a direct curl call — include it in an <a href> or window.location redirect.
# "id_token_hint" is strongly recommended: it helps Okta identify which session
# to terminate without prompting the user for confirmation.
5. Introspect a Token (Active / Inactive Check)
curl 
curl -X POST https://dev-123456.okta.com/oauth2/default/v1/introspect \
  -u "client_id:client_secret" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "token=<access_or_id_token>&token_type_hint=id_token"

# Response when active:
# { "active": true, "sub": "00u...", "exp": 1661293224, ... }
#
# Response when expired or revoked:
# { "active": false }
#
# Use introspection for opaque tokens or when you need a definitive
# server-side active/inactive check rather than local JWT validation.
6. Exchange Authorization Code for Tokens
curl 
curl -X POST https://dev-123456.okta.com/oauth2/default/v1/token \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=authorization_code" \
  -d "code=SplxlOBeZQQYbYS6WxSbIA" \
  -d "redirect_uri=https://app.example.com/callback" \
  -d "client_id=0oa48e74ox4t7mQJX0g7" \
  -d "code_verifier=abc123_your_pkce_verifier_here"

# Returns: access_token, id_token, refresh_token, token_type, expires_in, scope
# For public clients (SPA, native app): use code_verifier (PKCE) instead of client_secret.
# For confidential clients: add -u "client_id:client_secret" instead of code_verifier.
All Okta endpoint URLs are discoverable from the /.well-known/openid-configuration document — never hardcode them. When working with multiple authorization servers or custom domains, always construct the discovery URL from the issuer and let the library handle the rest.
