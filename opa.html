The Secrets Problem for AI Agents

AI agents are not passive scripts — they call APIs, query databases, write files, and trigger external workflows. Each action requires a credential. The problem: most organizations treat agent credentials the same way they treated service account passwords in 2012. That is a serious security debt.

⚠ Verizon DBIR: 80% of Breaches Involve Compromised Credentials
Non-human identities (service accounts, API keys, bots) are the fastest-growing attack surface in enterprise environments. Agents amplify the risk because they hold credentials for dozens of downstream systems simultaneously.

Hardcoded Secrets

API keys embedded in environment variables, config files, or source code are static — they never rotate, never expire, and are frequently over-shared. One leaked .env file exposes everything the agent can reach.

Credential Sprawl

A single agent framework may hold keys for OpenAI, Salesforce, Jira, Slack, and three internal APIs. Multiply by 1,000 agents and 10 tools each — that is 10,000 credentials to track, rotate, and audit. No spreadsheet survives this at scale.

No Audit Trail

Static API keys provide no attribution. If an agent calls Salesforce with a shared service account password, the log shows the service account — not which agent, which task, or on whose behalf. Forensics become impossible.

No Revocation Path

When a compromised agent is shut down, its secrets live on — in other agents that inherited the same config, in CI/CD pipelines, in backups. Without a central vault, "revoking" access means hunting across dozens of systems manually.

Scale Reality Check
Enterprise agentic deployments already reach thousands of agent instances. At 1,000 agents × 10 tool integrations each, an organization manages 10,000 live credentials with no central governance — unless they have a secrets vault. OPA is that vault, built natively into the Okta identity fabric.

What "Proper" Agent Secrets Management Looks Like

The requirements are non-negotiable for any enterprise-grade agentic deployment:

Requirement Why It Matters Without It
Central vault Single place to manage, rotate, and audit all credentials Credentials scattered across env vars, k8s secrets, config files
Just-in-time issuance Agent receives credential only when executing a specific task Long-lived credentials persist between tasks, expanding blast radius
Automatic rotation Passwords and API keys rotate on schedule without agent code changes Credentials age indefinitely; rotation requires coordinated downtime
Full audit log Every credential checkout is logged with agent ID, task, and delegating user Breaches are uninvestigable; no attribution beyond service account name
Identity-linked access Credential access is gated on agent identity, not just possession of a vault token Any process with vault access can read any secret

How OPA Works for Agents

Okta Privileged Access is a PAM (Privileged Access Management) product built directly into the Okta platform. For AI agents, it provides three core capabilities: secrets vaulting, just-in-time credential brokering, and automatic rotation — all tied to agent identity through the Okta token exchange flow.

🔒 Secrets Vaulting

API keys, service account passwords, database connection strings, and TLS certificates stored in OPA's encrypted vault. Agents never hold credentials in memory beyond the duration of a single task.

⏱ JIT Credential Brokering

An agent presents its Okta access token, OPA verifies identity and policy, and issues a time-limited credential — checked out for the task, checked in automatically on expiry.

🔄 Automatic Rotation

OPA discovers servers (via ISPM or manual configuration), schedules password rotation, and updates the vault without any agent code changes. Zero-downtime rotation for service accounts.

📺 Session Recording

SSH and RDP sessions brokered through OPA are recorded. For agent-to-server access, the full command log is stored for forensics, compliance, and anomaly detection by ITP.

Three Credential Types via Token Exchange (RFC 8693)

When an agent needs a secret, it performs a token exchange at the Okta authorization server. The exchange produces one of three credential types depending on what the downstream resource requires:

Credential Type Flow Output Use Case
Authorization Server ID-JAG assertion → token exchange → access token OAuth 2.0 access token (JWT) Agent calls another OAuth-protected API (Salesforce, internal microservice)
Secret (Static) Okta access token → OPA credential checkout API key, password string, certificate Agent calls legacy API or database requiring static credentials
Service Account Okta access token → OPA + Universal Directory Service account username + rotated password Agent authenticates to Windows servers, Active Directory, or LDAP-bound systems

Architecture: Agent → OPA → Time-Limited Secret → Resource

The agent authenticates to OPA using its own Okta access token (issued via Client Credentials or ID-JAG). OPA validates the token signature, issuer (https://tenant.okta.com/oauth2/<authServerId>), and audience. The token carries agent_id, task_id, and actor (delegating user) as custom claims.
2
→ OPA evaluates access policy — does this agent have permission to this secret? OPA → Policy Engine
OPA's policy engine checks: (a) Is the agent's identity registered and active in Universal Directory? (b) Does the agent's group membership grant access to the requested secret? (c) Is the request within allowed time windows? (d) Has the agent already exceeded its checkout quota? Policy is managed in the OPA Admin Console under Managed Connections.
3
→ OPA retrieves the credential from the encrypted vault OPA → Vault
OPA fetches the current credential from its internal encrypted store. For service accounts with rotation enabled, OPA may first rotate the password, then return the new value. The credential is decrypted in memory — it is never stored in plaintext at rest.
4
→ OPA returns a time-limited credential to the agent OPA → Agent
The credential is returned with an explicit TTL (e.g., 1 hour). OPA logs the checkout event to the Okta System Log: agent identity, task context, secret name, timestamp, and delegating user. The agent uses the credential immediately for its task.
5
→ Agent calls target resource; credential auto-expires after TTL Agent → Resource
The agent uses the credential to authenticate to the target resource (database, API, server). When the TTL expires, the credential is invalid — no explicit check-in is required. For session-based access (SSH/RDP), OPA terminates the session automatically and the recording is stored. If the agent is revoked mid-task, Universal Logout invalidates the OPA session immediately.

Zero Standing Privileges (ZSP)

Zero Standing Privileges is the security principle that no identity — human or machine — should hold persistent access to privileged resources. Agents amplify the value of ZSP: a compromised agent with a static API key is a persistent breach; a compromised agent with no stored credentials is a contained incident.

ZSP Defined for AI Agents
An agent operating under ZSP has zero credentials stored anywhere at rest. It requests a credential at task start, uses it for the duration of the task, and the credential expires automatically. If the agent is shut down, compromised, or cloned — the attacker gets nothing. There are no secrets to steal.

Traditional Credentials vs OPA Zero Standing Privileges

✗ Traditional: Static API Key

  • Stored in environment variable or config file
  • Same key used for every API call, every task
  • Expiry: 365 days (or never)
  • Rotation: manual, requires agent restart
  • If leaked: full API access until manually revoked
  • Audit: no per-agent attribution — just key hash
  • Blast radius: any process with env access can use it

✓ OPA: Dynamic Secret Checkout

  • Never stored at rest — fetched at task start
  • Unique checkout per task (can be scoped per request)
  • Expiry: 1 hour (configurable down to minutes)
  • Rotation: automatic, zero-downtime
  • If leaked: expires within TTL; revocable instantly
  • Audit: full System Log entry with agent ID, task, actor
  • Blast radius: single task, single agent, single TTL window

How OPA Implements ZSP for Agents

Credential Checkout + Auto Check-In

Agents check out credentials for a defined TTL. There is no manual check-in required. When the TTL expires, the credential is invalidated server-side — even if the agent is still running.

Session Recording

For infrastructure access (SSH, RDP, database consoles), OPA records the session. Every command an agent runs on a server is logged, searchable, and available for anomaly detection via ITP.

Just-in-Time Elevation

Certain privileged operations (e.g., production database write access) require real-time approval. OPA supports JIT elevation workflows — a human approves the request before the credential is issued, with a tight TTL after approval.

🔎 Attacker Perspective
Under ZSP, an attacker who compromises an AI agent binary gets: the agent's access token (expires in 1 hour), the in-memory credential for the current task (expires at task end), and nothing else. They cannot move laterally to other systems the agent is authorized to reach — they would need to issue a new OPA credential checkout, which requires a valid Okta token and triggers an anomaly alert in ITP.

OPA vs CyberArk vs HashiCorp Vault

Customers rarely come to OPA as a greenfield PAM decision. Most have CyberArk for privileged human access, HashiCorp Vault for infrastructure secrets, or both. The conversation is always "why OPA too?" — not "why OPA instead." Here is how to frame it.

Key Framing: Three Different Trust Anchors
OPA is identity-native — it runs inside Okta's security fabric, and every access decision is rooted in a verified Okta identity. CyberArk is vault-native — a standalone PAM platform optimized for human privileged access and safe-vaulting. HashiCorp Vault is infra-native — a DevOps-first secret store optimized for programmatic access by infrastructure tooling (Terraform, Kubernetes, CI/CD). They solve related but distinct problems.

Integration with the Okta Identity Fabric

OPA is not a standalone vault — it is a layer within Okta's full identity fabric. Every OPA event is visible in the Okta System Log; every agent identity is a Universal Directory entity; every access policy can reference Okta groups and FGA relationships. The integrations below make OPA meaningfully more powerful than a standalone PAM product.

ISPM → OPA

Okta Identity Security Posture Management continuously discovers non-human identities across connected SaaS and infrastructure systems. When ISPM detects a service account or API key that is not associated with any OPA-managed agent identity, it flags it as an orphaned credential — a potential shadow agent or stale access.

What ISPM Finds

  • API keys in GitHub, Bitbucket, or CI/CD systems not in OPA
  • Service accounts in Active Directory with no OPA checkout history
  • Credentials with no last-used date — dormant, removable

OIG → OPA

Okta Identity Governance runs access certification campaigns. For OPA-vaulted secrets, OIG can include credential access reviews: "Does agent svc-agent-crm-001 still need access to the Salesforce production API key?" Reviewers see the last checkout date, the delegating user, and the business justification.

What OIG Certifies

  • Quarterly review of agent-to-secret mappings
  • Automated revocation for overdue certifications
  • Segregation of duties checks on privileged secret access

ITP → OPA

Identity Threat Protection ingests OPA credential checkout events from the System Log. Anomaly signals: checkout volume spike (agent checking out credentials at 10x normal rate), off-hours access, access from an unusual network, or checkout of a secret never previously accessed.

ITP Responses

  • Suspend agent's Okta session (blocks future checkouts)
  • Trigger CIBA step-up — require human approval for next checkout
  • Alert SOC with full checkout history context

Universal Logout → OPA

When a threat is confirmed — compromised agent, insider threat, or runaway automation — Universal Logout revokes all active Okta sessions for the agent identity simultaneously. OPA sessions tied to the invalidated token are terminated immediately. In-flight tasks using OPA credentials fail cleanly.

Revocation Scope

  • All active OPA credential checkouts invalidated
  • All OAuth tokens for the agent revoked
  • All active SSH/RDP sessions terminated

Managed Connections in the Admin Console

IT administrators configure which agent identities can access which secrets through the OPA Managed Connections interface in the Okta Admin Console. This is the operational control plane — no code required for policy changes.

SE Demo Script

This is the recommended five-step demo flow for OPA AI agent secrets management. Total demo time: 12–15 minutes. Works best after you have already covered Agent Identity so the audience understands agent registration in Universal Directory.

1

Show the Agent Registered in Universal Directory

Navigate to Admin Console → Directory → People (switch to non-human entities view). Show the agent service principal: svc-agent-crm-001. Point out the agent_type attribute, assigned groups (ai-agents-crm-prod), and lifecycle state (Active).

Talking Point "Every agent that touches OPA-managed secrets must first exist as a verified identity in Universal Directory. This is the foundation — you cannot vault secrets for an identity that doesn't exist. This prevents shadow agents from ever getting credentials."
2

Navigate to Managed Connections — Show Configured Secret Access

Navigate to Admin Console → Privileged Access → Managed Connections. Show the agent's allowed secret list: Salesforce production API key (1-hour TTL, 90-day rotation), PostgreSQL CRM write password (30-minute TTL, 7-day rotation), Slack webhook (24-hour TTL).

Talking Point "This is IT's control plane. No secret access exists unless it is configured here. The TTLs are per-credential and per-agent — the CRM database gets a 30-minute window because write access is higher risk. Any engineer can see exactly what each agent can reach, and change it instantly without touching agent code."
3

Trigger a Token Exchange Request — Show OPA Returning a Time-Limited Credential

Use the demo CLI or Postman to send a token exchange request from the agent's client credentials. Show the raw HTTP exchange: agent presents access token, OPA returns the vaulted API key with an expires_at timestamp. Highlight the response headers showing the TTL.

Show the System Log Entry — Who Accessed What, When, On Whose Behalf

Navigate to Admin Console → Reports → System Log. Filter by the agent's entity ID or the event type system.agent.credential.checkout. Show the log entry: agent identity, secret name, checkout timestamp, TTL, and — critically — the delegating user (actor: jane.doe@company.com) from the OBO delegation chain.

Talking Point "This is the audit trail that CyberArk cannot provide for agent workloads. We know it was this agent, checking out this secret, at this time, on behalf of this human. 5

Revoke Access — Show Instant Credential Invalidation

In the Managed Connections interface, remove the Salesforce key from the agent's allowed secrets. Then attempt a new checkout request from the agent. Show the 403 Forbidden response with error: access_denied. Navigate back to System Log to show the revocation event.

Talking Point "This is the 'break glass' scenario. The agent is acting anomalously — maybe ITP flagged it, maybe a developer made a mistake, maybe you discovered a credential leak. IT removes access here, and the next checkout attempt fails instantly. Any credentials already issued expire at their TTL — within the hour. Compare that to revoking a static API key from 47 places in your config management system."

Objection Handling

🗣 "We already have CyberArk. Why would we add OPA?"

Agree first: "CyberArk is excellent for privileged session management for humans. The PSM product in particular is mature and we respect it. We're not here to replace CyberArk."

Identify the gap: "CyberArk manages privileged access for humans. AI agents are a fundamentally different identity class — they're non-human, they operate at machine speed, they need RFC 8693 token exchange, and they need a delegation chain that says 'this agent acted on behalf of Jane Doe.' CyberArk has no concept of an Okta agent identity or an OBO actor claim."

Position OPA: "OPA fills the identity-native gap. Keep CyberArk for human PAM. Add OPA for agent secrets — it's a different problem space and a different buyer in the security org."

🗣 "We use HashiCorp Vault. Can't agents just use that?"

Agree first: "HashiCorp Vault is a great infrastructure secret store. If your Terraform, Kubernetes, and CI/CD pipelines use Vault, keep them there."

Identify the gap: "Vault's trust anchor is the infrastructure entity — Kubernetes service account, AppRole, AWS IAM role. For AI agents, the trust anchor is the Okta identity, not the infrastructure. When an agent acts on behalf of a human user, the audit trail needs to show that delegation chain. Vault has no concept of OBO delegation, no RFC 8693 token exchange, and no CIBA step-up for JIT approval."

Better Together position: "Vault manages infra secrets; OPA governs identity-layer secrets. Okta is the authority for both. An agent can check out infrastructure credentials via Vault (using its Okta-issued Kubernetes service account token) and identity-layer secrets via OPA (using its Okta access token). You get the best of both — and a single audit plane in the Okta System Log."

Current Limitations — Honest Assessment

These are the constraints SE teams and customers need to plan around. Being upfront here builds credibility — these are known gaps on the roadmap, not surprises.

Licensing Dependency

Agent credential management via OPA requires both an OPA license and either an ISPM or AI license. OPA alone is not sufficient for the full agent secrets management capability set.

Billing Unit

The Workload Identity Resource Unit is the billing unit for vaulted credentials. Each agent identity with active Managed Connections counts against this unit. Factor this into SKU conversations with procurement.

JIT Checkout & Rotation Tier

Just-in-time credential checkout and automatic secret rotation are available, but require a specific OPA tier. Confirm tier eligibility before scoping these capabilities in a proof-of-concept.

OIG Integration — Phase 2

Integration with the AI agent lifecycle via OIG access certifications (periodic review of agent-to-secret mappings, automated revocation for overdue certifications) is Phase 2 with a GA target of FY27. This is the governance closure loop — currently requires manual OIG campaign setup.

SE Talking Point for Limitations
"The core vault capabilities — secrets management, JIT credential brokering, automatic rotation, and the full audit trail — are available today. The OIG integration for agent lifecycle governance closes the loop in Phase 2. For customers with immediate compliance requirements around agent access reviews, the interim answer is manual OIG campaign configuration targeting the agent's Okta entity."
