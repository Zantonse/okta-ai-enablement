Architecture Deep-Dive

The unified identity stack creates a seamless flow from user authentication through agent authorization to resource access—with every hop auditable and policy-controlled.

COMPLETE IDENTITY FLOW FOR AI AGENTS
1. User → Authenticates to GenAI App via Auth0 Universal Login
2. Auth0 → Issues ID token + Access token (federated IdP)
3. GenAI App → Stores tokens in Auth0 Token Vault (automatic)
4. Agent → Requests token for third-party API (e.g., Google Workspace)

10. Auth0 → Issues scoped access token to agent
11. Okta → Logs every hop: user auth, token exchange, action approval

Cross-App Access (XAA)

Enterprise-managed token exchange protocol for agent-to-resource delegation

▼

What It Is

Enterprise-managed token exchange protocol. Okta sits as the identity broker between requesting apps/agents and resource apps. Uses ID-JAG (Identity Assertion JWT) to carry user context through agent-to-resource interactions.

Sequence Diagram

XAA TOKEN EXCHANGE FLOW
1. User → Authenticates to Agent App via Okta SSO
2. Okta → Issues ID token to Agent App
3. Agent App → Needs to access Resource App on user's behalf
4. Agent App → Requests ID-JAG from Okta (presents ID token + agent identity)
5. Okta → Validates: agent registered? user assigned? policy met?
6. Okta → Issues ID-JAG (short-lived JWT with user + agent context)
7. Agent App → Presents ID-JAG + client credentials to Resource App
8. Resource App → Validates ID-JAG, issues scoped access token
9. Agent App → Accesses Resource App API with scoped token

Key Technical Details

  • Based on: OAuth 2.0 Token Exchange (RFC 8693)
  • ID-JAG payload: Contains sub (user), azp (agent app), aud (resource), scope, exp (short TTL)
  • Requirements: Agent registered in Okta Universal Directory, user-agent assignment, auth policy
  • Token chain: ID Token → ID-JAG → Scoped Access Token (each step validated)

Objection Handling

"We can just use service accounts" Service accounts have static creds, no user context, no audit trail. XAA preserves user identity through the chain—every agent action is attributed to a user.
"Microsoft Entra does token exchange too" Entra's token exchange is intra-Microsoft only. XAA works across any SaaS ecosystem—Google, Salesforce, AWS, custom apps.
"This adds latency" ID-JAG issuance is less than 50ms. Caching strategies available. Minimal tradeoff for complete auditability.

Token Vault

Secure storage for third-party OAuth tokens with automatic refresh

▼

What It Is

Auth0's secure storage for third-party OAuth tokens. Agents never touch raw tokens. Auth0 handles storage, refresh, and exchange. Eliminates token sprawl.

Sequence Diagram

TOKEN VAULT FLOW
1. User → Logs into GenAI App via Auth0 Universal Login
2. Auth0 → Authenticates and connects to upstream IdP (Google, GitHub, Slack)
3. Upstream IdP → Returns access token and refresh token
4. Auth0 → Stores tokens encrypted in Token Vault (automatic)
5. Agent → Requests token for third-party API (e.g., Google Calendar)
6. Auth0 → Checks validity. If expired, auto-refresh
7. Auth0 → Returns fresh, scoped access token
8. Agent → Calls third-party API

Key Technical Details

  • Federated Token Exchange: Auth0 mediates between agent and upstream IdPs
  • Automatic token refresh: No custom OAuth logic needed
  • Scoping: Tokens scoped per-provider, per-user, per-connection
  • CIAL: Client-Initiated Account Linking for adding provider connections post-login

  // Get access token for Google social connection

    connection: 'google-oauth2'
  });
  // Create Google OAuth client
  const googleOAuthClient = new google.auth.OAuth2();
  googleOAuthClient.setCredentials({ access_token: token });
  return googleOAuthClient;

Async Authorization (CIBA)

Human-in-the-loop approval for sensitive agent actions

▼

What It Is

CIBA enables human-in-the-loop approval for sensitive agent actions. Agent requests authorization, user gets push notification, approves or denies, agent proceeds. No browser redirect needed.

Sequence Diagram

CIBA APPROVAL FLOW
1. AI Agent → Needs to perform sensitive action (buy stock, transfer funds)
2. Agent → Sends CIBA request to Auth0: POST /bc-authorize
3. Auth0 → Sends push notification to user's device
4. User → Reviews context and approves or denies
5. Auth0 → Notifies agent via CIBA ping mode (webhook callback)
6. Auth0 → If approved: issues access token with requested scopes
7. Agent → Executes the action

Key Technical Details

  • Based on: OpenID Connect CIBA Core 1.0 and OAuth 2.0 RAR (RFC 9396)
  • Ping mode: Auth0 calls agent's callback URL when user responds (preferred)
  • Poll mode: Agent polls token endpoint (fallback)
  • Binding message: Human-readable string — "Do you want to buy N shares of TICKER?"
  • RAR: Rich Authorization Requests carry structured data (action, amount, resource)
  • Timeout: Configurable expiry (default 300s), agent gets expired_token error
  • Fallback: Secure link-based approval for users without push

Code Snippet

const auth0AI = new Auth0AI();

export const buy = (context: Context) => {
  const withAsyncAuthorization = auth0AI.withAsyncAuthorization({
    userID: context.userID,
    bindingMessage: async ({ ticker, qty }) =>
      `Do you want to buy ${qty} shares of ${ticker}?`,
    scopes: ["openid", "stock:trade"],
    audience: process.env["AUDIENCE"]!,
    onAuthorizationRequest: "block",
    onUnauthorized: async (e: Error) => {
      if (e instanceof AccessDeniedInterrupt) {
        return "The user has denied the request";
      }
      return e.message;
    },
  });
};

Objection Handling

"We can just use a confirmation dialog" Agents run in background. No UI. CIBA works across channels—SMS, push, email—and persists decisions across agent restarts.
"What about timeout?" Configurable. Clean error response, can retry or escalate. Timeout doesn't block the agent forever—it gets a clear signal.
"How is this different from MFA?" MFA verifies identity. CIBA authorizes a specific action with rich context. Different stage of the flow, different purpose.

FGA for RAG

Relationship-based access control for RAG pipelines

▼

What It Is

Auth0 FGA enforces relationship-based access control at retrieval time in RAG pipelines. Before an LLM sees a document, FGA checks: "Does this user have permission to view this document?"

RAG Pipeline Flow

FGA-PROTECTED RAG FLOW
1. User → Asks: "Show my last appointment record"
2. Agent → Sends query to vector store retriever
3. Vector store → Returns candidate documents (semantic similarity)
4. FGA Retriever → Intercepts: checks auth for each doc
5. FGA → Evaluates relationship graph
6. FGA Retriever → Filters unauthorized documents out
7. LLM → Generates answer using only authorized context

Key Technical Details

ReBAC Model Example:

model schema 1.1

type user

type patient
  relations
    define _self: [user]

type ai_agent
  relations
    define authorized_by: [patient]

type medical_record
  relations
    define can_update: owner
    define can_view: owner or delegated
    define delegated: [ai_agent#authorized_by]
    define owner: [patient]
  • Tuples: (user:john_doe, medical_record:mr_333, owner)
  • FGARetriever: SDK component wrapping vector store and FGA check
  • Performance: Sub-millisecond authorization checks at scale
  • Compatibility: Works with LangChain, Vercel AI, LlamaIndex

const documents = await readDocuments();

const vectorStore = await MemoryVectorStore.fromDocuments(
  documents,
  new OpenAIEmbeddings({ model: "text-embedding-3-small" })
);

const retriever = FGARetriever.create({
  retriever: vectorStore.asRetriever(),

    user: `user:${user}`,

    relation: "viewer",

Objection Handling

"We use RBAC already" RBAC is too coarse for document-level access. FGA operates at object-level with relationship graphs—who owns what, who can delegate, etc.
"This will slow down retrieval" Sub-millisecond checks. Batch available. Post-generation filtering is worse—LLM already saw the data.
"We'll build our own" FGA is open standard (OpenFGA). Auth0's implementation is managed, scaled, and integrates natively with RAG frameworks.

Agent Discovery & Governance

Complete lifecycle management for AI agents

▼

What It Is

Okta's platform for discovering shadow AI agents, registering them as managed identities, enforcing policies, and monitoring behavior.

Four Phases

1. Discover

  • ISPM: Scans for non-human identities including AI agents

Competitive War Room

Head-to-head comparison with leading competitors and tactical responses to common objections.

Microsoft Entra
Has:
Basic token exchange within Microsoft ecosystem, Entra ID Governance
Lacks:
Cross-ecosystem XAA, agent-native identity types, CIBA for agents, FGA/ReBAC
Key Gap:

Token exchange only works within Microsoft's ecosystem. Enterprise uses multi-cloud and multi-SaaS.

"We have Copilot Studio" Copilot Studio builds agents. Okta governs ALL agents, not just Microsoft's. Any agent. Any cloud. Any SaaS.
"Entra does token exchange" Only within Microsoft's ecosystem. XAA works across any OAuth-compliant resource—Google, Salesforce, AWS, custom apps, your legacy API.
Ping Identity
Has:
Strong legacy IAM, API security (PingGateway)
Lacks:
Agent-native discovery, no token vault, no FGA, limited async auth
Key Gap:

Built for human users, not AI agents. Ping doesn't know what an AI agent is.

"PingOne is cloud-native" Cloud-native IAM does not equal agent-native IAM. Where's agent discovery? Token vault? FGA for RAG?
Astrix Security
Has:
Good ISPM/NHI discovery
Lacks:
Everything after discovery - no auth, no tokens, no governance, no SDKs
Key Gap:

Detection without enforcement. They find the agent. Then what?

"We found 10,000 non-human identities" Great. Now what? Okta discovers AND manages the full lifecycle—registration, policies, tokens, audit. Astrix stops at discovery.
ServiceNow
Has:
IT workflow automation, ITSM integration
Lacks:
Identity-native controls, no token exchange, no FGA, no CIBA
Key Gap:

IT workflow tool, not identity platform. Different problem.

"We manage AI agent workflows" Workflow does not equal identity. Who authenticates the agent? Who governs its access? How do you audit? ServiceNow is for ops; Okta is for identity.
CyberArk
Has:
✓ Strong PAM install base, secrets vaulting, session recording, zero standing privileges (ZSP) messaging, dedicated AI agent security positioning
Lacks:
— No OIDC token issuance, no OBO/ID-JAG token exchange, no CIBA, no FGA/ReBAC, no cross-SaaS federation, no MCP authorization
Key Gap:

CyberArk controls the vault. Okta governs the identity federation, token issuance, delegation chain, and cross-SaaS authorization. CyberArk knows what secret the agent has; Okta knows who authorized the agent, on whose behalf, with what scope, and for how long.

"We already have CyberArk for PAM" CyberArk vaults secrets. Okta governs delegation. When an agent needs to access Salesforce on behalf of Bob in marketing, CyberArk can give it a credential — but can it prove Bob authorized it, scope it to read-only, expire it in 5 minutes, and log the delegation chain? That's XAA + ID-JAG + CIBA. They're complementary layers.
"CyberArk has zero standing privileges for agents" ZSP for secrets is good. But agent security isn't just secrets — it's delegation authority. An agent with zero standing secrets but permanent delegation scope is still over-privileged. Okta provides just-in-time delegation (ID-JAG with 5-minute TTL) on top of just-in-time secrets.

Troubleshooting & Edge Cases

Token Expiration During Long-Running Tasks

▼

Problem: Agent is running a long task and token expires mid-execution.

Solution:

  • Use Token Vault auto-refresh: Auth0 handles token refresh automatically.
  • For XAA, ID-JAG has short TTL—re-request from Okta if expired.
  • Implement retry logic in agent code to catch TokenExpiredError.

FGA Rule Conflicts

▼

Problem: Overlapping FGA rules causing unexpected access denials.

Solution:

  • Use FGA playground to visualize: fga-cli model test
  • Check tuple ordering — more specific rules first.
  • Test with check API: fga.check({ user, object, relation })

CIBA Timeout Tuning

▼

Problem: CIBA timeout too short; users don't have time to approve.

Solution:

  • Default: 300s (5 min). Adjustable per flow.
  • Financial services: 600-900s recommended.
  • Fallback: link-based approval if user doesn't have push.

Multi-Tenant Agent Isolation

▼

Problem: Agent in one tenant can see data from another tenant.

Solution:

  • Use Auth0 Organizations for tenant separation.
  • Okta: Use unique agent identity per tenant, with org-scoped policies.
  • FGA: Tenant as relationship dimension: (agent:ai-123, tenant:acme, assigned_to)

Known Limitations (EA)

Current constraints and workarounds for EA features. Use these when customers raise edge cases or blockers.

Cross-App Access (XAA) EA

▼
  • Separate trust domains required — The requesting app IdP and the resource app Authorization Server must be different issuers. Same-org XAA is not supported.
  • OIN registration required — Both apps must be registered in the Okta Integration Network (OIN). Custom OIDC apps cannot yet use Manage Connections for XAA.
  • First-party/internal app support — Not yet available. First-party app support is on the roadmap.
  • Immutable sub claim in ID-JAG — The sub claim is the Okta user ID and cannot be remapped to email or other attributes.
  • No custom claims in ID-JAG — Custom claims are not supported in ID-JAG tokens at this time.
  • SAML not supported as subject_token — Only OAuth/OIDC ID tokens can be used as the subject token in the exchange.
  • Auth0 XAA Beta: 5 RPS rate limit — The /token endpoint for ID-JAG exchanges has a 5 requests-per-second rate limit.
  • Auth0 XAA: Confidential client required — The requesting app must be a confidential client and must be first-party.
  • Auth0 XAA: One XAA-enabled connection per issuer — Only one XAA-enabled connection per upstream IdP issuer is allowed.

Token Vault EA

▼
  • Disable refresh token rotation — Applications using Token Vault must have refresh token rotation disabled. Rotation breaks the stored token lifecycle management.
  • MFA "Always" policy breaks token exchanges — If MFA is set to "Always" required, token exchanges will fail. Set the policy to "Never" and use Auth0 Actions to enforce selective MFA where needed.
  • Provider refresh tokens expire after 1 year — Third-party provider refresh tokens (Google, Slack, etc.) are automatically deleted after 1 year of non-use.
  • Connected Accounts consent per user per provider — Each user must grant consent individually for each provider. There is no admin-level pre-consent for connected accounts.

CIBA (Async Authorization) EA

▼
  • Enterprise Plan required — CIBA is only available on Okta Enterprise plan orgs.
  • Guardian enrollment or verified email required — Users must have Okta Verify/Guardian enrolled or a verified email to receive CIBA push notifications.
  • binding_message max 64 characters — The binding message shown to users during approval is limited to 64 characters.
  • Cannot modify RAR in Actions for CIBA flows — Rich Authorization Request (RAR) objects cannot be modified via Auth0 Actions during CIBA flows.
  • on_authorization_request="block" not for production — This mode holds the HTTP thread open indefinitely and will cause timeouts. Do not use in production.
  • Production pattern — Use GraphResumer with LangGraph interruptions for production-grade CIBA flows. This avoids blocking threads.

Fine-Grained Authorization (FGA) EA

▼
  • BatchCheck limit: 50 items per server call — The FGA BatchCheck API is limited to 50 items per call. The SDK automatically splits larger batches into multiple calls.
  • No built-in result caching — FGA does not cache check results between calls. Implement application-level caching if repeated checks are needed in hot paths.
  • handleToolErrors must be false in LangGraph ToolNode — When using FGA with LangGraph's ToolNode, set handleToolErrors: false so FGA authorization errors propagate correctly rather than being silently swallowed.
  • FGAFilter (Vercel) has no asJoinedStringTool() — The Vercel AI SDK FGAFilter does not include the asJoinedStringTool() helper. Manual wrapping of the filtered result is required.

ISPM (Agent Discovery) EA

▼
  • SAM plugin is Chrome-only — The SAM Browser Plugin requires managed Chrome deployed via Google Admin Console. Other browsers are not supported.
  • 7-day initial data lag — After deploying the SAM plugin, expect a 7-day delay before meaningful discovery data appears in ISPM dashboards.
  • ISPM is reactive, not preventive — ISPM discovers existing OAuth grants and shadow AI usage. It does not block new grants at the point of authorization.
