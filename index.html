The Problem

Meet Your Next
Insider Threat

Anyone can spin up an AI agent in minutes. These agents bypass SSO, skip MFA, and hold direct access to your most sensitive data. Traditional security was built for humans.

🛡

Shadow AI Sprawl

Users build agents with no approval process. IT can see who logs in, but what the agent does after login is completely invisible.

🔑

Over-Permissioned Accounts

Agents are configured with service accounts that grant full admin access, long-lived tokens, and no user binding.

👤

Missing User Context

Agents operate without user context. They need to act on behalf of a user with permissions lesser than or equal to that user's.

📊

No Audit Trail

Agents work 24/7 at 5,000 ops/minute. EU AI Act Article 14 mandates every agent action be logged with who, why, when, and on whose behalf.

Auth0 for AI Agents

The Developer
Security Toolkit

A suite of identity features that help developers build secure GenAI applications. SDKs for every major AI framework.

🔒

User Authentication

Securely authenticate users with Universal Login — OAuth 2.0 and OpenID Connect for both interactive and headless AI agents. Support for social, enterprise, and custom identity providers.

🗄

Token Vault

Securely connect agents to external tools — Google, Slack, GitHub. Manages obtaining, storing, and refreshing API tokens. Agents never handle sensitive credentials directly.

🛑

Async Auth (CIBA)

Human-in-the-loop for critical actions. Agents request approval via push, SMS, or email using Client-Initiated Backchannel Authentication. Supports Rich Authorization Requests (RAR).

🔍

Authorization for RAG

Enforce document-level access control in RAG pipelines using Auth0 FGA. Only data the user is authorized to see is included in LLM context. Prevents sensitive data leakage.

context-docs.ts — FGA-protected RAG tool
import { FGARetriever } from "@auth0/ai-langchain/RAG";

const retriever = FGARetriever.create({
  retriever: vectorStore.asRetriever(),
  // FGA tuple: check if user can view this document
  buildQuery: (doc) => ({
    user: `user:${user?.email}`,
    object: `doc:${doc.metadata.documentId}`,
    relation: "can_view",
  }),
});

// Only authorized documents reach the LLM
const documents = await retriever.invoke(question);

Architecture

Reference Patterns

Five foundational patterns for securing agent-to-resource and agent-to-agent communication, grounded in OpenID Foundation standards.

Pattern A — XAA

Agent-to-Resource (Enterprise)

Agent accesses internal APIs (HR, CRM, ERP) on behalf of authenticated users.

Agent Auth (PKI) → XAA Token Exchange → ID-JAG → Resource Access
Pattern B — Token Vault

Agent-to-Resource (Consent)

Agent accesses third-party SaaS (Gmail, Salesforce, Slack) with user consent.

OAuth Consent → Token Vault Store → Auto Refresh → SaaS API
Pattern C — A2A

Agent-to-Agent

Multiple specialized agents collaborate, with delegation chains preserved.

Agent A (Client) → Validate Identity → Agent B (Resource)
Pattern D — CIBA

Human-in-the-Loop

Sensitive actions (large transactions, PII access) require explicit human approval.

Agent Request → CIBA Push → User Approves → Token Issued
Pattern E — FGA + RAG

RAG Security

AI agent uses RAG with enterprise documents — access levels enforced at retrieval time.

RAG Query → FGA Check → Filtered Context → LLM Response
Technical Deep-Dive

Token Exchange Flow

How an AI agent securely obtains credentials to access protected resources on behalf of an authenticated user via Okta's org authorization server.

1

User Authenticates

User authenticates with the Okta org authorization server using a web app (Authorization Code + PKCE). Server returns an ID token to the web app.

2

ID Token Passed to Agent

The web app passes the ID token to the AI agent so it can perform actions on the user's behalf. Every subsequent action is bound to this user context.

3

Agent Exchanges Token

Agent sends the ID token to Okta and requests exchange for resource credentials (ID-JAG, secret, or service account). Okta validates against Managed Connections configuration.

4

ID-JAG to Access Token

If the credential is an ID-JAG (Identity Assertion JWT), the agent sends it to the custom authorization server to exchange for a usable, scoped access token.

5

Resource Access

Agent uses the access token to request the protected resource. Every action is logged: who, what, when, on whose behalf, with a digital signature.

The Four Layers Framework

CIOs think in four layers when evaluating AI. Align your conversation to all four — not just technology:

  • People — What new skills and teams do I need? I can't spin up new teams overnight.
  • Process — AI changes how domains connect. My SaaS-aligned org structure may not work anymore.
  • Technology — Table stakes. Has to run. But it's just one layer.
  • Data — Your AI strategy is only as good as your data strategy.
"If you're not talking about those layers to your CIO, you're just talking about a technology. Then you are just a connector in a Martech slide — a single-source vendor for one thing."— CIO, Fortune 500

Discovery

36 Discovery Questions

Organized by category. Use during initial pitch, pre-workshop, or qualification calls.

Business & AI Strategy (6 questions) ▼
  1. What are your top 3 business drivers for adopting AI agents? (efficiency, innovation, customer experience, cost reduction)
  2. Which business functions are currently piloting or deploying AI agents? (engineering, support, sales, operations, finance)
  3. What is your current AI governance maturity level? (ad-hoc, defined, managed, optimized)
  4. How do you currently measure AI risk and trustworthiness metrics?
  5. Who owns the AI security budget and decision authority?
  6. Are there board-level or regulatory mandates driving AI security investments?
AI Architecture & Integration (6 questions) ▼
  1. What AI/LLM platforms are in production or planned? (OpenAI, Anthropic, Azure, Bedrock, Vertex, local models)
  2. What agent frameworks are you using? (LangChain, LangGraph, AutoGen, Semantic Kernel, CrewAI, custom)
  3. Are you implementing RAG patterns? What vector databases and retrieval systems?
  4. How many distinct AI agents are currently deployed or in development?
  5. Do agents operate autonomously, on behalf of users, or both?
  6. What MCP servers or tools are agents accessing (first party, 3rd party, both)?
Authentication & Authorization (6 questions) ▼
  1. How do AI agents authenticate today? (API keys, service accounts, OAuth client credentials, certificates)
  2. Are credentials long-lived or short-lived? How are they rotated?
  3. How granular is your current authorization model? (application-level, API-level, data-level)
  4. Do you need to track delegation chains (which user an agent acts on behalf of)?
  5. Are there scenarios requiring real-time human approval before agent actions?
  6. How do you currently handle agent access to third-party SaaS APIs?
Agent Governance & Lifecycle (6 questions) ▼
  1. Do you have a registry or inventory of all deployed AI agents?
  2. How are new agents onboarded and existing agents decommissioned?
  3. Are there known instances of shadow AI or unsanctioned LLM usage?
  4. How do you govern agent access to sensitive data classifications?
  5. What compliance frameworks apply? (SOC2, HIPAA, GDPR, industry-specific)
  6. Do you perform access reviews or certifications for agent permissions?
Security & Threat Protection (6 questions) ▼
  1. How do you handle privileged access for agents performing admin actions?
  2. Do you monitor for anomalous agent behavior or token misuse?
  3. Are AI agent activities integrated into your SIEM/SOAR pipelines?
  4. Do you have visibility into what data agents are accessing and exfiltrating?
  5. Would you benefit from risk scoring or adaptive access for AI interactions?
  6. How do you detect and respond to anomalous agent behavior or unauthorized access attempts?
PoC Readiness & Success Criteria (6 questions) ▼
  1. Which AI use case would be most impactful to secure first?
  2. What environments can we use for PoC testing? (sandbox, staging, production clone)
  3. Who will participate in the PoC? (AI developers, identity architects, security ops, compliance)
  4. What KPIs define success? (time to deploy, risk reduction, compliance coverage, developer experience)
  5. Are there upcoming milestones to align with? (audits, board reviews, fiscal deadlines)
  6. What is the path from successful PoC to production deployment?
