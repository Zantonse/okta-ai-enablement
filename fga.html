The Problem with Coarse Authorization

Why OAuth scopes are insufficient for AI agents, the scenarios where they break down, and the confused deputy problem.

OAuth Scopes Are Flat Strings

OAuth scopes are designed as coarse-grained permission signals. They are human-authored strings that an authorization server validates at the API boundary:

OAuth 
scope: "read:files write:files calendar:read"

These scope strings tell an API "this client is allowed to read files." They do not express:

⌛What OAuth Scopes Cannot Express
Required Constraint OAuth Scope Can Express? FGA Can Express?
Which specific files No Yes — per object
Which rows in a database No Yes — per resource instance
Which tools an AI agent may invoke No Yes — per tool per user
Which document a RAG pipeline may retrieve No Yes — pre-retrieval filtering
Only /Marketing/Q1 Plans/ folder in Google Drive No — Drive scope is all-or-nothing Yes — folder-level tuple
The Core Problem For human-driven applications with predictable, deterministic behavior, coarse scopes are often sufficient. The human sees the consent screen and implicitly understands they are granting access to their own files. For AI agents, none of those implicit constraints hold. Agents act autonomously, at machine speed, across data the user may not have consciously inventoried.

What Specifically Goes Wrong Without FGA

Without fine-grained authorization at the resource layer, AI agents operating on broad OAuth tokens create predictable failure modes:

Without FGA
Sales AI queries company docs
Returns confidential M&A memos and unreleased forecast data if vector search similarity matches the query — regardless of the user's clearance level.
With FGA
Sales AI queries company docs
Only returns documents where the user's role has an explicit viewer or editor relationship tuple. M&A docs are invisible to the retrieval pipeline.
Without FGA
Code review bot
Has access to the entire organization's repositories via the developer's inherited GitHub token. A prompt injection attack on the bot has blast radius of all repos.
With FGA
Code review bot
Scoped to repo:billing-service, read-only. FGA check at every tool invocation enforces this ceiling regardless of what credentials are in the environment.
Without FGA
Healthcare AI reads patient records
Can access all records indexed in the vector database. Any user interacting with the AI could surface records they have no clinical relationship to.
With FGA
Healthcare AI reads patient records
Filtered to only records where the doctor#attending_physician relationship tuple exists for the requesting clinician.

The Confused Deputy Problem in AI Agent Chains

The confused deputy is a class of privilege escalation where a less-privileged caller tricks a more-privileged service into using its authority on the caller's behalf. In AI agent systems, this is not theoretical — it is a structural property of how agents inherit ambient authority.

Confused Deputy in an AI Agent Chain
Developer (broad GitHub token — all repos)
    |
    v
AI Coding Assistant  ←── [inherits developer token]
    |
    v
Sub-agent: Security Scanner  ←── [inherits assistant ambient authority]
    |
    v
Sub-agent now has full repo access it was never intended to have
    |
    v
Prompt injection attack on security scanner = blast radius of ALL repos
Attack Vector Any sub-agent in a chain — including specialized agents the user never directly interacts with — inherits the ambient authority of its parent. A prompt injection into any node in the chain can escalate to the full credential scope. This is not a bug in the agent; it is a structural property of ambient authority without per-resource enforcement.

RBAC faces "role explosion" as AI use cases multiply. Organizations that start with admin, editor, viewer inevitably accumulate:

The Zanzibar Model

Google's 2019 foundational paper on consistent, global authorization — and the design patterns it introduced that underpin every modern FGA system.

The 2019 Google Paper

In 2019, Google published "Zanzibar: Google's Consistent, Global Authorization System" at USENIX ATC. This was significant because Google rarely shares core infrastructure internals — but recognized that authorization at scale is a universal problem the industry was solving badly.

10M+ Permission checks per second
2T Access control lists managed
<10ms p95 check latency
99.999% Uptime over 3 years
1,500+ Namespaces across Google services
Google Services Using Zanzibar Gmail, Google Drive, Google Calendar, YouTube, Google Cloud, Maps, Google Photos — all authorization decisions across all of these flow through Zanzibar. The system is not a prototype; it is production infrastructure at the largest scale any authorization system has ever operated at.

Relationship Tuples: The Core Data Unit

The fundamental data unit in Zanzibar is the relationship tuple — a triple of (object, relation, user) written in the format:

Tuple Format 
object#relation@user

Reading each component: object is the resource being protected, relation is the access type, user is who holds the relationship. Real examples:

document:readme#owner@alice
folder:projects#viewer@bob
group:engineers#member@charlie

The most powerful feature is usersets — instead of a specific user, the subject can be a set of users defined by a relation on another object:

document:readme#viewer@group:engineers#member

This single tuple grants viewer access to every current and future member of the engineers group — no per-user enumeration required, no stale list to maintain.

Why Usersets Matter for AI When an AI agent runs as a service account on behalf of users, usersets let you write one tuple — "the agent can view documents that the user can view" — rather than enumerating every document the user has access to. The relationship graph handles the transitive resolution.

Namespace Configurations and Set Operations

While tuples store raw relationship data, namespace configurations define how relationships combine to produce permissions. These use set operations:

Zanzibar Namespace Config 
document namespace:
  "viewer" = direct_viewers
             UNION editors
             UNION (viewers of parent_folder)

  "editor" = direct_editors
             UNION owner

The available set operations are:

Operation Meaning Example Use Case
Union User has permission if they satisfy any of the component sets viewer = direct viewers OR editors OR folder viewers
Intersection User has permission only if they satisfy all of the component sets can_export = has_export_role AND is_account_owner
Exclusion User has permission from set A but NOT from set B can_view = all_employees EXCEPT blocked_users

The consequence: when Bob becomes an editor, he automatically gains viewer rights through the union. When the parent folder grants viewer access to the Marketing group, every Marketing member gets document viewer access — without any additional tuples being written per document.

The "New Enemy Problem" and Zookies

One of Zanzibar's key innovations solves the new enemy problem — a timing-based security vulnerability that naively-cached authorization systems are susceptible to:

New Enemy Problem Timeline
t=0  Alice removes Bob from a shared document
t=1  Alice immediately adds highly sensitive content to that document
t=2  Authorization system serves a cached/stale check result
t=2  Bob reads the newly added sensitive content  ← SECURITY BREACH

ZANZIBAR SOLUTION: Zookies (consistency tokens)
t=0  Alice removes Bob → write operation returns a zookie
t=1  Alice adds content with the zookie as a consistency requirement
t=2  Check request: "use data no older than this zookie"
t=2  Stale cache results are rejected → Bob correctly denied

Zookies are opaque timestamps that ensure authorization checks use data fresh enough relative to the content being authorized. Applications specify consistency requirements per-request, trading latency for freshness based on the sensitivity of the operation.

The Three Core APIs

POSTCheck API

"Does user X have permission Y on object Z?"

Input: (user, relation, object)
Output: { "allowed": true | false }

Primary enforcement point. Called at every access decision. Sub-10ms at p95. The most performance-critical API in the system.
POSTExpand API

"Who are all the entities with permission Y on object Z?"

Input: (relation, object)
Output: Full permission tree / userset expansion

Used for auditing and debugging — shows exactly which relationship paths grant access, including all transitive paths through groups and hierarchies.
POSTListObjects API

"What objects of type T does user X have permission Y on?"

Input: (user, relation, type)
Output: [list of authorized objects]

Critical for RAG pipelines: returns all document IDs the user can view before querying the vector database. Prevents unauthorized content from entering the LLM context window.
ListObjects Is the RAG Enabler ListObjects is the API that makes FGA practical for AI document retrieval. Instead of querying the vector DB and then checking each result, you first call ListObjects to get the authorized document set, then restrict your vector search to that set. Unauthorized documents are never retrieved.

The Authorization Model DSL

OpenFGA provides a human-readable DSL (Domain-Specific Language) compiled to JSON for the API. The DSL defines types, relations, and permissions — the schema of your authorization model.

Basic Document Model

OpenFGA DSL 
model
  schema 1.1

type user

type document
  relations
    define owner: [user]
    define editor: [user]
    define viewer: [user, user:*]

  permissions
    define can_view   = viewer or editor or owner
    define can_edit   = editor or owner
    define can_delete = owner

Full Model with Teams, Folders, and Conditions

OpenFGA DSL — Full Example 
model
  schema 1.1

type user

type team
  relations
    define member: [user]

type folder
  relations
    define owner:  [user]
    define editor: [user, team#member]
    define viewer: [user, team#member, user:*]

  permissions
    define can_view = viewer or editor or owner
    define can_edit = editor or owner

type document
  relations
    define owner:  [user]
    define editor: [user, team#member]
    define viewer: [user, team#member, user:*]
    define parent: [folder]

  permissions
    define can_view   = viewer or editor or owner
                        or viewer from parent    # folder hierarchy
    define can_edit   = editor or owner
    define can_delete = owner
    define can_share  = owner

OpenFGA DSL — Conditions 
model

Contextual Tuples (Per-Request Context)

Contextual tuples are relationship data passed in the check request itself — they exist only for the duration of that single request and are never stored. This enables authorization decisions based on runtime context:

🕐Contextual Tuple Use Cases
  • Time-based access: "it is currently 14:00 on a Tuesday — within business hours"
  • Network context: "request originated from IP 10.0.0.1 — inside the corporate network"
  • Session-level MFA: "user completed MFA step-up in this session"
  • Geographic restrictions: "user is accessing from the EU region"
HTTP — Contextual Tuple in Check Request 
{
  "tuple_key": {
    "user": "user:alice",
    "relation": "can_execute",
    "object": "mcp_tool:send_email"
  },
  "contextual_tuples": {
    "tuple_keys": [
      {
        "user": "user:alice",
        "relation": "has_mfa_active",
        "object": "session:current"
      }
    ]
  }
}

The contextual tuple asserts "alice has an active MFA session" for the duration of this check only. The authorization model can require this as a condition for sensitive operations without storing session state in FGA permanently.

FGA for MCP Tool Authorization

Applying the Zanzibar model to MCP server tool access — per-user per-tool grants, middleware patterns, and the AuthZen standard for standardized authorization decisions.

The Problem: MCP Tools Are Overpowered by Default

MCP (Model Context Protocol) servers expose tools that AI agents invoke. Without authorization enforcement at the tool layer, any connected agent — or any user of that agent — can call any tool. This violates the principle of least privilege at the most critical point in an AI system: the interface between the language model and real-world actions.

Threat Model A single MCP server may expose tools that read files, send email, execute code, query databases, make API calls, and modify calendar entries. Without per-tool per-user authorization checks, the entire tool surface is accessible to any session that connects to the server — including sessions compromised by prompt injection.

Authorization Model for MCP Tools

OpenFGA DSL — MCP Tool Model 
model
  schema 1.1

type user

type organization
  relations
    define member: [user]
    define admin:  [user]

type mcp_tool
  relations
    define executor:    [user, role#assignee]
    define owner:       [user]
    define belongs_to:  [organization]

  permissions
    define can_execute = executor
                         or owner
                         or member from belongs_to

type role
  relations
    define assignee: [user]

type conversation
  relations
    define owner:  [user]
    define viewer: [user]
    define editor: [user]

  permissions
    define can_view = viewer or owner
    define can_edit = editor or owner

This model enables three authorization paths for a tool:

  • Direct grant: user is directly assigned executor on a specific tool
  • Role-based: user is an assignee of a role that has executor rights
  • Organization-based: user is a member of an org that owns the tool

MCP Server Middleware Pattern

The FGA check runs as middleware before every tool handler executes. The user identity is extracted from the JWT (issued by Okta), and the check is performed against Auth0 FGA. The tool handler only executes on allowed: true.

Python — MCP Middleware 
from openfga_sdk.client import OpenFgaClient, ClientConfiguration
from openfga_sdk.client.models import ClientCheckRequest

fga_client = OpenFgaClient(ClientConfiguration(
    api_url=os.environ.get("FGA_API_URL"),
    store_id=os.environ.get("FGA_STORE_ID"),
    authorization_model_id=os.environ.get("FGA_MODEL_ID")
))

async def authorize_tool_call(user_id: str, tool_name: str) -> bool:
    """FGA middleware check — runs before every MCP tool invocation."""
    response = await fga_client.check(
        ClientCheckRequest(
            user=f"user:{user_id}",
            relation="can_execute",
            object=f"mcp_tool:{tool_name}",
        )
    )
    return response.allowed

# In MCP tool handler:
@mcp_server.tool("search_code")
async def search_code_tool(query: str, ctx: MCPContext):
    user_id = ctx.get_user_id()  # From JWT sub claim

    if not await authorize_tool_call(user_id, "search_code"):
        raise PermissionError(
            f"User {user_id} is not authorized to use search_code"
        )

    # ... execute tool logic

The AuthZen Pattern for Standardized Authorization Decisions

The AuthZen standard provides a standardized REST API for making authorization decisions — decoupling the enforcement point (the MCP server) from the policy decision point (Auth0 FGA or any compliant engine). This is the pattern recommended for enterprise MCP deployments where the authorization engine may need to be swapped or audited independently.

AuthZen — HTTP 
POST /access/v1/evaluation

{
  "subject": {
    "type": "user",
    "id": "alice"
  },
  "action": {
    "name": "execute"
  },
  "resource": {
    "type": "mcp_tool",
    "id": "send_email"
  },
  "context": {
    "current_hour": 14,
    "day_of_week": "Tuesday",
    "user_mfa_verified": true
  }
}

// Response:
{
  "decision": true,
  "decision_id": "abc-123-audit-trail"
}
The decision_id Is Your Audit Trail Every AuthZen response includes a decision_id — an immutable identifier for this specific authorization decision at this moment in time. Every AI agent action can be traced back to the authorization decision that permitted it. This is the mechanism that satisfies GDPR's right-of-access documentation and HIPAA's audit log requirements for AI agent actions.

FGA for RAG / Document Security

5.1 The Core Risk: OWASP GenAI LLM02 — Sensitive Information Disclosure

OWASP GenAI Top 10 — LLM02 Risk

Sensitive Information Disclosure is listed as a critical vulnerability in the OWASP GenAI Top 10 2025. Without authorization enforcement at the retrieval layer, an AI assistant can surface confidential data to any user whose query vector happens to be semantically similar to a restricted document.

Concrete examples of unauthorized disclosure in production RAG systems:

User Query Without FGA (RAG returns) With FGA (RAG returns)
"What is the Q4 forecast for ZEKO?" EXPOSED Financial forecast (unauthorized employee) FILTERED Only docs the user's FGA tuples permit
"Show me recent patient labs for John" EXPOSED Patient record (non-treating physician) FILTERED Only records where attending_physician relationship exists
"What are our competitor pricing notes?" EXPOSED Competitive intel to wrong sales team FILTERED Only shown to SE / permitted role via group tuple
"Summarize the M&A due diligence" EXPOSED Legal docs outside attorney-client privilege FILTERED Legal docs visible only to privileged#member users
"There has been a 1,025% increase in AI-related vulnerabilities (CVEs) in a single year — and 77.4% of those are directly tied to API security issues." — Forrester, 2025

5.2 Two Filtering Approaches

Approach 1 (Recommended)
Pre-Retrieval Filtering

Call ListObjects first to get the set of document IDs the user is authorized to see. Then pass that allowlist as a filter constraint to the vector DB query.

Strongest security

Unauthorized documents never enter the embedding search. One additional FGA call before retrieval.

Approach 2 (Acceptable)
Post-Retrieval Filtering

Query the vector DB normally — get top-K results. Then run BatchCheck against FGA for each result in parallel. Strip unauthorized documents before sending context to the LLM.

Good coverage

This is the pattern FGARetriever uses. The LLM never sees unauthorized content, but the vector DB did process the full corpus.

Key Point

In both approaches the LLM context window is clean — no unauthorized content is ever passed as prompt context. The difference is whether unauthorized document vectors are even evaluated during search.

5.3 FGARetriever — Drop-in Authorization for LangChain / LlamaIndex

Auth0 provides the FGARetriever class via the @auth0/ai-langchain package. It wraps any LangChain-compatible retriever and transparently applies FGA post-retrieval filtering.

JavaScript / LangChain — FGARetriever

JavaScript 
import { FGARetriever } from '@auth0/ai-langchain';
import { ClientBatchCheckItem } from '@openfga/sdk';

// user is the authenticated principal from the JWT (e.g. decoded from Okta token)
const retriever = FGARetriever.create({
  retriever: vectorStore.asRetriever(),
  buildQuery: (doc) => ({
    user:     `user:${user.email}`,
    object:   `doc:${doc.metadata.documentId}`,
    relation: 'can_view',
  }),
});

// FGARetriever workflow:
//  1. Calls vectorStore.asRetriever() → semantically relevant docs (top-K)
//  2. Calls FGA BatchCheck for each doc in parallel
//  3. Returns only docs where allowed === true
//  4. Unauthorized docs are silently stripped — never reach the LLM
const documents = await retriever.invoke(query);

from auth0_ai_langchain import FGARetriever
from openfga_sdk.client.models import ClientBatchCheckItem

@tool
def get_context_documents_tool(query: str):
    """Retrieve documents the current user is authorized to view."""
    retriever = FGARetriever(
        retriever=vector_store.as_retriever(),
        build_query=lambda doc: ClientBatchCheckItem(
            user=f"user:{user_email}",
            object=f"doc:{doc.metadata.get('documentId')}",
            relation="can_view",
        ),
    )
    documents = retriever.invoke(query)
    return "\n\n".join([document.page_content for document in documents])
Full RAG + FGA Post-Retrieval Flow
  1. User query is embedded into a vector and sent to the vector database
  2. Vector DB returns top-K semantically similar documents (e.g. doc:public-overview, doc:q4-forecast, doc:competitor-analysis)
  3. FGA BatchCheck runs in parallel for all results — user:alice can_view doc:q4-forecast returns false (alice not in finance group)
  4. Only authorized documents (doc:public-overview, doc:competitor-analysis) are included in the LLM context window
  5. LLM generates a response from filtered context — doc:q4-forecast content is never seen

6.2 When to Use Which

Is your use case Kubernetes admission, CI/CD, or infrastructure policy?

    → OPA (Rego) — general purpose, sidecar-friendly, already integrated into K8s ecosystem


Are your permissions based on relationships between users and resources?
(document sharing, folder hierarchies, multi-tenant isolation, AI agent tool gating)

    → OpenFGA / Auth0 FGA — purpose-built, native ListObjects API, FGARetriever for RAG


Are you in an AWS-native environment and need formally verifiable, human-readable policies?

    → Cedar — readable DSL, mathematical proofs, AWS Verified Permissions integration


Do you need relationship-based checks AND attribute/context conditions?

    → OpenFGA with Conditions (CEL expressions) — e.g. "within business hours," "MFA is active," "amount < $1000"


Do you need to express specific transaction consent in an OAuth flow?

    → OAuth RAR (RFC 9396) — use as complement to FGA, not a replacement

Rich Authorization Requests (RFC 9396) extends OAuth to include rich, structured permission objects in the authorization request. Example: a payment consent screen specifying a specific amount and merchant.

Mechanism Role Use When
OAuth RAR (RFC 9396) Communicates specific transaction scope intent to the resource server at consent time. It does not enforce — the receiving API must validate and act on the RAR object Consent screen for a specific transaction (payment, data export). Expressing rich context about what the client wants to do

The Relationship

RAR and FGA are complementary, not competing. RAR captures consent intent in the token; FGA enforces fine-grained access to the actual resource. For AI agents calling multiple APIs with complex shared-resource semantics, use both: RAR for the OAuth consent flow, FGA for the authorization check at every resource access.

7.2 Seeding FGA from Okta Groups via Event Hooks

Okta groups can be synchronized into FGA relationship tuples via Okta Event Hooks. When a user joins or leaves a group in Okta Workforce, an event fires and the corresponding tuple is written or deleted in Auth0 FGA.

Python 
# Okta Event Hook handler — fires on group membership changes
async def handle_okta_group_event(event: dict):
    user_id  = event["data"]["userId"]    # e.g. "00u1234"
    group_id = event["data"]["groupId"]   # e.g. "00g5678"
    action   = event["eventType"]         # "group.user_membership.add" | "...remove"

    fga_group = f"group:{group_id}"
    fga_user  = f"user:{user_id}"
    tuple_key = {
        "user":     fga_user,
        "relation": "member",
        "object":   fga_group,
    }

    if action == "group.user_membership.add":
        # Write tuple to Auth0 FGA
        await fga_client.write({
            "writes": { "tuple_keys": [tuple_key] }
        })

    elif action == "group.user_membership.remove":
        # Delete tuple from Auth0 FGA — access revoked immediately
        await fga_client.write({
            "deletes": { "tuple_keys": [tuple_key] }
        })

7.5 XAA — Cross-App Access Protocol for Agent Delegation

Auth0 is implementing the XAA (Cross-App Access) open protocol, which extends OAuth 2.0 to support secure agent-to-app and app-to-app communication. This solves the confused deputy problem at the protocol layer.

XAA Delegation Flow
  1. User interacts with App A (the requesting app / AI agent frontend)
  2. App A needs to call App B (a resource app — e.g. a document API, calendar service) on the user's behalf
  3. XAA exchanges an ID-JAG (identity assertion) token for a scoped access token with only the permissions App B requires
  4. The Identity Provider (Okta) is the consent authority — the user sees and approves the delegation in one place
  5. User gets centralized visibility into all agent connections from the Okta portal — one location to review and revoke

Centralized Consent Okta manages all agent delegation consents in one portal Users see every agent they've authorized — no shadow approvals
Scoped Tokens Each agent-to-app leg gets a minimum-privilege token Eliminates the confused deputy problem — sub-agents can't inherit excessive permissions
Revocation Governance team or user can revoke any agent's access from one place Instant blast-radius containment if an agent is compromised
Audit Trail Every delegation event is logged in Okta's immutable audit log GDPR / SOC 2 / HIPAA compliance for agentic AI access
