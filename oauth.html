Understanding OAuth 2.1

A ground-up guide to modern authorization — from core concepts through MCP server authentication and Okta’s implementation.

What is OAuth 2.1?

OAuth 2.1 (draft-ietf-oauth-v2-1-15, March 2026) is a security framework that lets users grant third-party applications limited access to their resources — without sharing their password. It consolidates RFC 6749 (OAuth 2.0) and RFC 6750 (Bearer Token Usage) into a single, modernized specification, removing flows that have proven insecure in practice and mandating best-practice extensions that were previously optional.

The core promise: OAuth solves delegated authorization — "I want App A to be able to read my data from Service B, but I don't want to give App A my Service B password." The user approves the delegation; the authorization server issues a token; the app uses that token. The user's credential never leaves Service B.

The Four Parties

Every OAuth 2.1 interaction involves four roles. Understanding who each party is — and what they trust — is the foundation for understanding every grant flow.

Resource Owner

The entity that owns the data being accessed — almost always the end user. The resource owner grants (or denies) permission when they click "Allow" on the consent screen. They are authorizing the client to act on their behalf within the defined scope.

Client

The application requesting access to protected resources. OAuth 2.1 draws a hard distinction between two client types, because they have fundamentally different security properties:

Client Type Can it keep a secret? Examples Key requirement
Confidential Yes — code runs on a server the user cannot inspect Traditional web app (Node/Rails/Django backend), server-to-server daemon Authenticates to the AS with a client secret or private key JWT
Public No — code runs where the user can read it Single-page app (React/Vue), native mobile app, CLI tool Must use PKCE; never issued a static client secret

Authorization Server (AS)

The trusted authority that authenticates the resource owner and issues tokens. In Okta, this is your org's authorization server (e.g., https://acme.okta.com/oauth2/default). The AS owns the consent screen, validates redirect URIs, and signs tokens. It exposes a set of standardized endpoints discoverable via the /.well-known/openid-configuration metadata document.

Resource Server (RS)

The API or backend service that holds the protected data. The RS receives Bearer tokens on each API request and validates them — either by calling the AS's introspection endpoint or by verifying the JWT signature locally using the AS's published JWKS (JSON Web Key Set). The RS never sees the user's credential; it only evaluates the token.

Tokens

OAuth 2.1 defines three token types that each serve a distinct purpose.

Access Token

The credential presented to the Resource Server. Access tokens are intentionally short-lived (typically 1–60 minutes). If an access token leaks, it expires quickly on its own, limiting the blast radius. OAuth 2.1 recommends sender-constraining access tokens (e.g., DPoP) so a stolen token cannot be replayed from a different client.

Security principle: An access token is a bearer credential — anyone who holds it can use it. Keep them short-lived and scope them narrowly. OAuth 2.1 section 7.1 recommends access tokens be valid for no more than the time needed to complete the operation.

Refresh Token

A long-lived credential issued alongside the access token that allows the client to obtain a new access token after the current one expires — without requiring the user to re-authenticate. Refresh tokens are issued only when the offline_access scope is requested and approved.

Rotation (mandatory for public clients in OAuth 2.1): Every time a public client uses a refresh token, the AS issues a brand-new refresh token and invalidates the old one. This is called refresh token rotation. If a leaked refresh token is ever used by an attacker, the legitimate client's next use of the old (now-invalidated) token will signal a theft — the AS should revoke the entire token family.

Refresh token reuse detection: If the AS receives a refresh token that has already been rotated (i.e., used and superseded), this is evidence of token theft. RFC 6749 left this optional; OAuth 2.1 makes rotation and reuse detection required for public clients.

ID Token (OIDC)

Not part of OAuth 2.1 itself, but issued by OpenID Connect — a thin identity layer built on top of OAuth 2.0/2.1. ID tokens are JWTs that assert who the user is (authentication), while access tokens assert what the client is allowed to do (authorization). An ID token must never be sent to an API as a Bearer token; it is only for the client application to consume locally.

JWT Anatomy

Access tokens and ID tokens are commonly issued as JSON Web Tokens (JWTs, pronounced "jot"). A JWT is three base64url-encoded JSON objects concatenated with dots:

Header {"alg":"RS256","typ":"JWT","kid":"v1"} Declares the signing algorithm and identifies which key to verify with (via kid)

Base64url is encoding, not encryption. Anyone can decode the header and payload of a JWT without a key. Never put sensitive data (PII, secrets) in a JWT payload unless the token is also encrypted (JWE). The signature only guarantees integrity and origin — not confidentiality.

Scopes and Claims

Scopes

Scopes are strings that represent a bundle of permissions the client is requesting. The user sees scopes on the consent screen. They travel in the authorization request (scope=openid profile email) and appear in the issued token so the Resource Server can enforce them.

Standard OIDC scopes: openid (required for OIDC; triggers ID token issuance), profile (name, locale, etc.), email, address, phone, offline_access (triggers refresh token issuance).

Custom API scopes are defined on the Authorization Server and represent specific API permissions, e.g., read:transactions, write:orders.

Claims

Claims are the key-value pairs inside the JWT payload that assert facts about the subject. A scope is the request; claims are the response. Requesting the profile scope causes the AS to include claims like name, given_name, family_name, and picture in the ID token.

Claims can be standard (defined in OIDC Core) or custom (added by your organization). Custom claims can encode roles, department, license tiers, or any attribute needed for authorization decisions at the Resource Server.

OAuth 2.0 to 2.1 — What Changed and Why

OAuth 2.1 is not a new protocol — it is a curated subset of OAuth 2.0 plus mandatory adoption of security extensions that were already widely deployed. The changes fall into two categories: removal of insecure flows, and promotion of optional best practices to requirements.

Removed: Implicit Grant (response_type=token)

The Implicit flow was designed in 2012 for single-page apps that couldn't make server-side token requests. Instead of an authorization code, it delivered the access token directly in the URL fragment after redirect.

Why it was removed:

  • URL fragment leakage: The fragment is visible in browser history, Referer headers, server logs, and to JavaScript on the page — including third-party analytics scripts.
  • No refresh tokens: SPAs had to re-run the implicit flow silently via hidden iframes — a pattern that breaks with ITP/ETP browser cookie restrictions.
  • Token injection attacks: Without PKCE, there is no way to bind the token to the specific client that initiated the flow.
  • Better alternative: Auth Code + PKCE works perfectly in SPAs and is now universal.

Removed: Resource Owner Password Credentials (ROPC)

ROPC allowed a client to collect the user's username and password directly and exchange them for tokens by POSTing to /token. It was designed as a "migration" flow for legacy apps.

Why it was removed:

  • Credential phishing: The user's password is exposed to the client — violating the core OAuth delegation model. A malicious or compromised client can harvest credentials.
  • MFA bypass: The flow happens entirely in a back-channel POST. There is no browser redirect, so there is no way to intercept the flow to challenge the user with MFA, step-up auth, or bot detection.
  • No consent screen: Users cannot see what they are approving. There is no way to express granular scope consent.
  • Better alternative: Auth Code + PKCE with a login widget provides a secure, MFA-capable, consent-driven flow even for first-party apps.

Mandated: PKCE for All Authorization Code Flows

PKCE (Proof Key for Code Exchange, RFC 7636) was originally designed to protect native apps from authorization code interception — a malicious app on the same device registering the same redirect URI. OAuth 2.1 requires PKCE for all clients, including confidential web apps, because it also provides protection against CSRF and code injection attacks even when the client secret is in play.

How PKCE works: Before the authorization request, the client generates a random code_verifier (43–128 chars). It derives a code_challenge by hashing the verifier with SHA-256 and base64url-encoding the result. The challenge goes into the authorization request. When exchanging the code for tokens, the client sends the original verifier. The AS hashes it and compares — if they match, the token request came from the same client that started the flow.

Mandated: Exact Redirect URI Matching

OAuth 2.0 allowed partial/pattern matching of redirect URIs. OAuth 2.1 requires the redirect URI in the token request to exactly match the one registered for the client — character for character, including trailing slashes. This eliminates open redirect vulnerabilities where an attacker registers https://evil.com as a path component and tricks the AS into redirecting the code there.

Prohibited: Tokens in Query Strings

OAuth 2.0 allowed access tokens to be passed as ?access_token=... query parameters. OAuth 2.1 prohibits this. Query parameters appear in server access logs, browser history, and HTTP Referer headers to third-party origins. Tokens must be transmitted in the Authorization: Bearer ... request header only.

Grant Flows

A grant flow (also called a grant type) is the specific sequence of requests and redirects by which a client obtains tokens. The right flow depends on what kind of client you are building and whether there is an interactive user present. Use this guide as a click-through reference — click any step to expand the HTTP details.

Choosing a flow: Is there a user? Yes — use Authorization Code + PKCE. No user, server-to-server only? Use Client Credentials. Device with no browser (TV, CLI, IoT)? Use Device Authorization. Need to refresh silently? Use the Refresh Token flow.

Authorization Code + PKCE

The primary flow for any application where a user is present. Works for web apps, SPAs, and native mobile apps. PKCE is mandatory in OAuth 2.1 regardless of whether the client is public or confidential.

Client App Authorization Server Resource Server
1 Generate PKCE code_verifier and code_challenge
// Client-side — never sent to server until step 5 // Step A: generate cryptographically random verifier code_verifier = base64url(random_bytes(32)) // e.g.: "dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk" // Step B: derive challenge code_challenge = base64url(sha256(ascii(code_verifier))) // e.g.: "E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM" // method used (always S256 in OAuth 2.1) code_challenge_method = "S256"

This step happens entirely on the client before any network request. The client generates a random code_verifier — a high-entropy random string between 43 and 128 characters (URL-safe base64 alphabet). It then SHA-256 hashes the verifier and base64url-encodes the result to produce the code_challenge.

The verifier is kept secret in memory. Only the challenge is sent to the AS in the next step. When the client proves knowledge of the verifier in step 5, the AS can verify it matches the challenge it stored — binding the token exchange to this specific flow initiation.

Plain method is prohibited in OAuth 2.1. code_challenge_method=plain (where the challenge equals the verifier verbatim) provides no protection and must not be used. Always use S256.
2 Redirect browser to /authorize with code_challenge
GET /oauth2/default/v1/authorize ?response_type=code &client_id=0oa1abc234DEF &redirect_uri=https%3A%2F%2Fapp.acme.com%2Fcallback &scope=openid%20profile%20email%20offline_access &state=xyzABCrandomCSRF123 &nonce=n-0S6_WzA2Mj &code_challenge=E9Melhoa2OwvFrEMTJguCHaoeK1t8URWbuGJSstw-cM &code_challenge_method=S256 HTTP/1.1 Host: acme.okta.com

The client instructs the browser to navigate to the AS's authorization endpoint. Key parameters:

  • response_type=code — requests an authorization code (not a token directly)
  • client_id — identifies the registered OAuth application
  • redirect_uri — where the AS will send the code back; must exactly match a registered URI
  • scope — space-separated requested permissions; offline_access requests a refresh token
  • state — random value the client generates to prevent CSRF; it will be echoed back in the redirect and the client must verify it matches
  • nonce — random value included in the ID token to prevent replay attacks (OIDC)
  • code_challenge + code_challenge_method — PKCE parameters
What OAuth handles vs. what it doesn't: The /authorize redirect is where authentication happens — the AS presents its login UI. OAuth itself doesn't define how users authenticate; that's OIDC and the AS's responsibility. OAuth only defines what happens after the user is authenticated.
3 User authenticates and grants consent
// User interaction at the Authorization Server: // 1. AS presents login page // 2. User enters credentials (password, MFA, passkey, etc.) // 3. AS presents consent screen listing requested scopes // 4. User clicks "Allow" // // The AS records: // - which user authenticated (sub) // - which client requested access (client_id) // - which scopes were approved // - the code_challenge received // - the redirect_uri received // // The AS generates a short-lived authorization code // (typically valid for 60–120 seconds, single use)

This step is entirely within the Authorization Server. The client application does not participate — the browser has navigated away to the AS's domain. This is a critical security property: the user's credential never touches the client application.

After successful authentication and consent, the AS creates an authorization code and associates it with the user, client, scopes, and PKCE challenge. The code is one-time-use and expires quickly to limit the window for interception.

First-party apps and consent: If your app is a first-party application (built by the same organization that owns the AS), the consent screen is often suppressed. OAuth 2.1 still runs the full flow — the consent is just pre-approved at the policy level. Do not skip PKCE because consent is suppressed.
4 AS redirects browser back with authorization code
HTTP/1.1 302 Found Location: https://app.acme.com/callback ?code=SplxlOBeZQQYbYS6WxSbIA &state=xyzABCrandomCSRF123

The AS instructs the browser to redirect back to the client's registered redirect_uri, appending the authorization code and the original state value as query parameters.

When the client's callback endpoint receives this request, it must:

  1. Verify that state matches the value it stored in step 2 — this prevents CSRF attacks
  2. Extract the code parameter
  3. Immediately exchange the code for tokens (step 5) before it expires
The code is not a token. The authorization code is a short-lived, single-use credential that can only be exchanged for tokens by a client that also holds the code_verifier. It is not useful to an attacker by itself (without the verifier, thanks to PKCE).
5 POST /token with code + code_verifier
POST /oauth2/default/v1/token HTTP/1.1 Host: acme.okta.com Content-Type: application/x-www-form-urlencoded Authorization: Basic BASE64(client_id:client_secret) // ^ for confidential clients only; omit for public clients grant_type=authorization_code &code=SplxlOBeZQQYbYS6WxSbIA &redirect_uri=https%3A%2F%2Fapp.acme.com%2Fcallback &code_verifier=dBjftJeZ4CVP-mB92K27uhbUJU1p1r_wW1gFWFOEjXk &client_id=0oa1abc234DEF // ^ include client_id for public clients (no secret)

This is a direct back-channel (server-to-server) POST from the client to the AS's token endpoint — the browser is not involved. The client proves possession of the authorization code and demonstrates it initiated this flow by providing the code_verifier.

The AS performs these validations before issuing tokens:

  1. Verify the code is valid, not expired, and not previously used
  2. Verify redirect_uri exactly matches what was in the authorization request
  3. Compute SHA-256(code_verifier) and compare to the stored code_challenge
  4. Authenticate the client (via Basic auth secret or client assertion JWT)
PKCE verification is the key step. Even if an attacker intercepted the authorization code from the redirect URL, they cannot exchange it without the code_verifier that only the legitimate client holds. This is why PKCE replaced the Implicit flow for SPAs.
6 Receive access_token, id_token, and refresh_token
HTTP/1.1 200 OK Content-Type: application/json Cache-Control: no-store { "token_type": "Bearer", "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6InYxIn0...", "expires_in": 3600, "scope": "openid profile email offline_access", "id_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6InYxIn0...", "refresh_token": "hGzdzs8Rv9NTrTNEwnbJUvLwTbC8vMWWKBX7PLJJ" }

On successful validation, the AS returns the token response. The client should:

  • Store the access_token in memory (not localStorage, not sessionStorage for SPAs) — it is used to call APIs
  • Store the refresh_token securely — for SPAs, this means a secure, HttpOnly cookie; for native apps, the platform keychain
  • Validate the id_token JWT signature and claims (iss, aud, exp, nonce) before trusting it
  • Note the expires_in value to know when to use the refresh token
Token storage for SPAs: Storing tokens in localStorage exposes them to XSS. The recommended pattern is to keep the access token in memory (JavaScript variable), use a secure HttpOnly cookie for the refresh token (managed by your backend or BFF), and re-acquire the access token via the refresh flow on page load.
7 Call API with Bearer access token
GET /api/v1/transactions HTTP/1.1 Host: api.acme.com Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6InYxIn0... --- // Resource Server validates the token: // 1. Decode JWT header to get kid (key ID) // 2. Fetch AS JWKS: GET https://acme.okta.com/oauth2/default/v1/keys // 3. Verify RS256 signature using matching public key // 4. Check exp (not expired), iss (matches AS), aud (matches this API) // 5. Check scp claim includes required scope for this endpoint // 6. Return response if all checks pass HTTP/1.1 200 OK Content-Type: application/json { "transactions": [...] }

The client attaches the access token to every API request in the Authorization header using the Bearer scheme. The Resource Server independently validates the JWT — it does not need to call the AS for each request if it can verify the signature locally.

Local JWT validation (preferred for performance): cache the AS's JWKS public keys and verify the signature, expiry, issuer, and audience on every request. Re-fetch JWKS only when a new kid appears that is not in the cache.

Audience validation is critical. The RS must verify that its own identifier appears in the aud claim. Without this check, a token issued for API A could be replayed against API B at the same organization — a confused deputy attack.

Client Credentials

For machine-to-machine (M2M) communication where there is no user. A backend service, cron job, or microservice obtains tokens using its own identity — no browser redirect, no consent screen. There is no resource owner in this flow; the client and resource owner are the same entity.

No PKCE in Client Credentials. PKCE protects the authorization code exchange in user-facing flows. Client Credentials has no authorization code and no redirect — the entire flow is a single back-channel POST. Security comes from keeping the client secret or private key confidential.
Client (Server) Authorization Server Resource Server
1 POST /token with client credentials
POST /oauth2/default/v1/token HTTP/1.1 Host: acme.okta.com Content-Type: application/x-www-form-urlencoded Authorization: Basic BASE64(client_id:client_secret) grant_type=client_credentials &scope=read%3Areports%20write%3Alogs // Alternative: private_key_jwt authentication (more secure) // grant_type=client_credentials // &client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer // &client_assertion=eyJ... (signed JWT assertion) // &scope=read:reports

The service POSTs directly to the token endpoint. It authenticates itself using one of:

  • HTTP Basic Auth (Authorization: Basic BASE64(id:secret)) — simple but requires protecting the secret at rest
  • client_secret_post — secret in the POST body (allowed but not preferred)
  • private_key_jwt (RFC 7523) — the client signs a JWT with its private key; the AS verifies with the registered public key. This is the most secure option because the secret never leaves the service.
Prefer private_key_jwt over client_secret. A client secret is a shared secret that can be exfiltrated from config or secret storage. A private key (stored in HSM or platform keystore) is computationally unfeasible to extract. Many enterprise AS implementations support private_key_jwt via JWKS URI registration.
2 Receive access_token
HTTP/1.1 200 OK Content-Type: application/json Cache-Control: no-store { "token_type": "Bearer", "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6InYxIn0...", "expires_in": 3600, "scope": "read:reports write:logs" } // Note: no id_token (no user) and no refresh_token (by design)

Client Credentials responses do not include an ID token (no user was authenticated) and typically do not include a refresh token. Refresh tokens are not needed here — if the access token expires, the service just runs the two-step flow again. Refresh tokens for M2M add attack surface without meaningful benefit.

Services should cache the access token and re-use it until it expires, then obtain a new one. Making a token request per API call is wasteful and will trigger rate limits on the AS.

3 Call API with Bearer token
GET /api/v1/reports/monthly HTTP/1.1 Host: api.acme.com Authorization: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6InYxIn0... HTTP/1.1 200 OK Content-Type: application/json { "report": { ... } }

Identical to step 7 of the Authorization Code flow. The Resource Server validates the token without knowing which grant type was used to obtain it — the JWT is the same structure either way. The sub claim in a Client Credentials token is the client_id, not a user ID.

Device Authorization (RFC 8628)

For devices that cannot open a browser or have constrained input (smart TVs, game consoles, CLI tools, IoT devices). The device displays a short code and URL; the user visits that URL on their phone or computer to approve access. The device polls the AS until authorization completes.

Device / CLI Authorization Server User's Phone / Browser
1 Device POSTs to /device_authorization
POST /oauth2/default/v1/device/authorize HTTP/1.1 Host: acme.okta.com Content-Type: application/x-www-form-urlencoded client_id=0oa1abc234DEF &scope=openid%20profile%20offline_access --- HTTP/1.1 200 OK Content-Type: application/json { "device_code": "GmRhmhcxhwAzkoEqiMEg_DnyEysNkuNhszIySk9eS", "user_code": "WDJB-MJHT", "verification_uri": "https://acme.okta.com/activate", "verification_uri_complete": "https://acme.okta.com/activate?user_code=WDJB-MJHT", "expires_in": 900, "interval": 5 }

The device makes a single POST to the device authorization endpoint. The AS responds with:

  • device_code — a long opaque code the device uses when polling; never shown to the user
  • user_code — a short, human-readable code (typically 8 chars) the user types at the verification URI
  • verification_uri — the URL the user visits (designed to be short enough to type)
  • verification_uri_complete — a URL with the code pre-filled (for QR codes)
  • interval — minimum seconds between polling attempts (the device must honor this)
  • expires_in — how long the user has to complete authorization
2 Device displays user_code and verification_uri
// Device output (terminal / screen): // // To sign in, visit: https://acme.okta.com/activate // Enter code: WDJB-MJHT // // Or scan QR code: [QR CODE IMAGE] // // Code expires in 15 minutes. // Waiting for authorization...

The device displays the user_code and verification_uri in a way the user can read. For TV apps this is on screen. For CLI tools this is printed to stdout. Some devices also display a QR code encoding the verification_uri_complete for convenience.

Meanwhile, the device begins polling the token endpoint (step 4) at the specified interval, waiting for the user to complete authorization on their phone.

3 User visits URI on their phone and authenticates
// User action on their phone/computer: // 1. Navigate to https://acme.okta.com/activate // 2. Enter "WDJB-MJHT" when prompted // 3. AS authenticates the user (login + MFA) // 4. AS shows consent screen for requested scopes // 5. User clicks "Allow" // 6. AS marks the device_code as authorized // // AS response to user's browser: // "Authorization granted. You may return to your device."

The user follows a completely normal authentication flow on their secondary device (phone, laptop). They may need to sign in, complete MFA, and approve the consent screen. The AS links their approval to the device_code so the polling device knows when to proceed.

This is a key security advantage: full MFA and consent even for headless devices.

The user_code is designed to be user-friendly. RFC 8628 specifies it should use characters that are unambiguous when read aloud or typed (no 0/O, 1/I/L confusion). The dash separator is visual grouping only and may be case-insensitive. WDJB-MJHT is an example of the recommended format.
4 Device polls /token with device_code
// Sent repeatedly at `interval` seconds until success or expiry: POST /oauth2/default/v1/token HTTP/1.1 Host: acme.okta.com Content-Type: application/x-www-form-urlencoded grant_type=urn%3Aietf%3Aparams%3Aoauth%3Agrant-type%3Adevice_code &device_code=GmRhmhcxhwAzkoEqiMEg_DnyEysNkuNhszIySk9eS &client_id=0oa1abc234DEF --- // While user hasn't acted yet: HTTP/1.1 400 Bad Request { "error": "authorization_pending", "error_description": "..." } // If polling too fast: HTTP/1.1 400 Bad Request { "error": "slow_down", "error_description": "..." } // Device must increase interval by 5 seconds when slow_down received // If user denied: HTTP/1.1 400 Bad Request { "error": "access_denied", "error_description": "..." } // If code expired: HTTP/1.1 400 Bad Request { "error": "expired_token", "error_description": "..." }

The device polls the token endpoint repeatedly. The AS returns authorization_pending until the user completes the flow. The device must respect the interval and back off when it receives slow_down. Polling too aggressively can result in the AS revoking the device code.

5 Receive access_token once user approves
// After user approves on their phone: HTTP/1.1 200 OK Content-Type: application/json { "token_type": "Bearer", "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6InYxIn0...", "expires_in": 3600, "scope": "openid profile offline_access", "id_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6InYxIn0...", "refresh_token": "hGzdzs8Rv9NTrTNEwnbJUvLwTbC8vMWWKBX7PLJJ" }

Once the user approves, the next polling request returns a successful token response identical to the Authorization Code flow. The device can now use the access token for API calls and store the refresh token to maintain the session without requiring the user to re-authorize.

Refresh Token

Used to obtain a new access token after the current one expires — without requiring the user to re-authenticate. This is not a standalone grant flow; it is used in conjunction with Authorization Code + PKCE or Device Authorization after the initial tokens have been issued.

Client App Authorization Server Resource Server
1 POST /token with refresh_token
POST /oauth2/default/v1/token HTTP/1.1 Host: acme.okta.com Content-Type: application/x-www-form-urlencoded grant_type=refresh_token &refresh_token=hGzdzs8Rv9NTrTNEwnbJUvLwTbC8vMWWKBX7PLJJ &client_id=0oa1abc234DEF &scope=openid%20profile%20offline_access // scope is optional; if omitted, AS issues same scopes as original grant // scope cannot be broader than the original grant

The client sends the refresh token to the AS's token endpoint. No browser redirect is required — this is a direct back-channel request. The client authenticates if it is a confidential client (Basic Auth or JWT assertion). Public clients include only their client_id.

The client can optionally request a narrower scope than the original grant. It cannot request broader scopes — attempting to do so results in an error.

Refresh token rotation (required for public clients in OAuth 2.1): The AS must issue a new refresh token on every use and invalidate the old one. If the AS receives a refresh token that it has already rotated (i.e., an old superseded token), this indicates the token was stolen. The AS should revoke the entire token family associated with this grant.
2 Receive new access_token + rotated refresh_token
HTTP/1.1 200 OK Content-Type: application/json Cache-Control: no-store { "token_type": "Bearer", "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6InYxIn0...", "expires_in": 3600, "scope": "openid profile offline_access", "id_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCIsImtpZCI6InYxIn0...", "refresh_token": "NEW_ROTATED_TOKEN_8Rv9NTrTNEwnbJUvLwTbC8vMWWKBX7" // ^ entirely new value — old refresh_token is now invalid }

The response contains a fresh access token, a new ID token, and — critically — a brand-new refresh token. The client must replace its stored refresh token with this new value immediately. Holding on to the old one is not useful; it is now permanently invalid.

If the AS does not return a new refresh token (allowed for confidential clients that use absolute expiry instead of rotation), the client continues to use the same refresh token until its absolute expiry or explicit revocation.

Absolute vs. idle expiry: Refresh tokens have two types of expiry. Absolute expiry (e.g., 24 hours from issuance) limits how long a session can last regardless of activity. Idle/inactivity expiry (e.g., 1 hour without use) logs the user out if they are inactive. Okta supports both independently and both are enforced even with rotation.

What Was Removed in OAuth 2.1

Two grant types present in OAuth 2.0 were removed from OAuth 2.1 because they cannot be made secure by any implementation measure. Their removal is not a configuration option — conforming OAuth 2.1 authorization servers must not support them.

Implicit Grant — response_type=token

How it worked: Instead of returning an authorization code to the redirect URI, the AS returned the access token directly in the URL fragment (#access_token=...). The intent was to avoid the extra round-trip needed for the code exchange, making it simpler for SPAs.

// OAuth 2.0 Implicit — NEVER USE GET /authorize ?response_type=token ← requests token directly &client_id=0oa1abc234DEF &redirect_uri=https://spa.example.com/callback &scope=openid profile // AS redirects to: // https://spa.example.com/callback#access_token=LEAKED_TOKEN&expires_in=3600 // ^^^^^^^^^^^^^^^^^^^^ in URL fragment

Why it is removed:

  • The access token appears in the browser history and is accessible to JavaScript on the page — including third-party scripts (analytics, A/B testing, chat widgets)
  • The Referer header can leak the fragment to third-party origins if any navigation follows the redirect
  • No refresh token is possible — silent iframe renewals break with modern browser ITP/ETP restrictions
  • No PKCE protection is possible — any app could register the same redirect URI and steal the token
  • Auth Code + PKCE has all the simplicity benefits of Implicit with none of the vulnerabilities

Resource Owner Password Credentials (ROPC) — grant_type=password

How it worked: The client collected the user's username and password directly (in its own login form) and POSTed them to the AS's token endpoint. The AS validated the credentials and returned tokens directly — no browser redirect, no consent screen.

// OAuth 2.0 ROPC — NEVER USE POST /oauth2/default/v1/token Content-Type: application/x-www-form-urlencoded grant_type=password ← removed in 2.1 &username=jane@acme.com ← user's credential in client's hands &password=hunter2 ← credential phishing risk &client_id=0oa1abc234DEF &scope=openid profile

Why it is removed:

  • Credential phishing: The user's password is typed into the client application — not into the AS's trusted login page. A compromised or malicious first-party app can harvest credentials for every user who logs in.
  • MFA is impossible: The entire exchange is a single back-channel POST. There is no way to inject a second-factor challenge, bot detection CAPTCHA, or step-up authentication into this flow. MFA requirements are silently bypassed.
  • No consent: There is no consent screen. Users cannot see or limit what the application is requesting.
  • Violates the delegation model: The core premise of OAuth is that the user's credential never leaves the AS. ROPC fundamentally breaks this guarantee.
  • Common misuse: ROPC was frequently used as a "shortcut" for first-party apps. The correct replacement is Auth Code + PKCE with an embedded web view (or system browser) pointed at the AS's own login page — users authenticate directly with the AS, and the app never sees the password.

CIMD — Client ID Metadata Documents

How CIMD Works

Instead of registering with an AS, the client uses a stable HTTPS URL as its client_id. That URL hosts a JSON metadata document. The AS fetches it on demand during authorization.

// Client hosts this at: https://app.example.com/oauth/client-metadata.json // This URL IS the client_id { "client_id": "https://app.example.com/oauth/client-metadata.json", "client_name": "Example MCP Client", "redirect_uris": ["http://127.0.0.1:3000/callback"], "grant_types": ["authorization_code"], "token_endpoint_auth_method": "none" } // AS advertises support via: // { "client_id_metadata_document_supported": true }

Advantages over DCR

  • No server-side state — AS fetches on demand, no database records accumulate
  • Domain-backed identity — TLS + domain ownership provides stronger identity signal
  • One URL per app — same client_id regardless of how many users or instances
  • No registration endpoint to attack — no DoS surface for database writes

Limitations

  • Requires the client to host a public HTTPS endpoint — not viable for local desktop apps or CLI tools
  • AS providers (Okta, Auth0, Ory) have not yet widely implemented CIMD support
Security: DCR Risks
Client impersonation: Any client can self-assert "client_name": "Claude Desktop" — the AS cannot verify identity without software statements.
State explosion: Every registration creates a permanent database record. Rate-limit the endpoint and implement cleanup policies.
DoS surface: An unauthenticated /register endpoint that writes to a database is trivially spammable.

OAuth Extensions

OAuth 2.1 provides the secure core. These companion specifications handle token validation, revocation, sender-constraining, and fine-grained authorization. Understanding them individually and as a system is essential for production deployments.

Token Introspection (RFC 7662)

What it does

A resource server calls the AS to check whether a token is valid: POST /introspect with the token, receive {"active": true/false, ...}. Essential when using opaque (non-JWT) tokens, or when real-time revocation status matters.

POST /oauth2/default/v1/introspect HTTP/1.1 Authorization: Basic BASE64(client_id:client_secret) Content-Type: application/x-www-form-urlencoded token=eyJhbGciOiJS... --- HTTP/1.1 200 OK { "active": true, "scope": "read write", "client_id": "0oa1abc", "sub": "user123", "exp": 1737275311, "token_type": "Bearer" }
Inactive/invalid tokens always return {"active": false} with HTTP 200 — never 4xx. This prevents information leakage about token existence.

Token Revocation (RFC 7009)

What it does

Immediately invalidate an access or refresh token. Always returns HTTP 200 (even for tokens that don’t exist — prevents info leakage).

POST /oauth2/default/v1/revoke HTTP/1.1 Authorization: Basic BASE64(client_id:client_secret) Content-Type: application/x-www-form-urlencoded token=hGzdzs8Rv9NTr...&token_type_hint=refresh_token
Cascade behavior: Revoking a refresh token invalidates all access tokens derived from it. Revoking only an access token leaves the refresh token alive.

Pushed Authorization Requests — PAR (RFC 9126)

What it does

Moves authorization request parameters from the browser URL (front-channel) to a server-to-server POST (back-channel). The AS stores the request and returns a request_uri. The browser redirect uses only client_id + request_uri.

Client Authorization Server
1POST /par with all authorization parameters
POST /oauth2/default/v1/par HTTP/1.1 Authorization: Basic BASE64(client_id:client_secret) Content-Type: application/x-www-form-urlencoded response_type=code&client_id=0oa1abc &redirect_uri=https://app.example.com/callback &scope=openid+accounts:read &code_challenge=E9Melhoa2Ow...&code_challenge_method=S256
All parameters sent back-channel. Client authenticates. AS validates immediately.
2Receive request_uri (valid ~60 seconds)
HTTP/1.1 201 Created { "request_uri": "urn:ietf:params:oauth:request_uri:bwc4JDEclQ0rz_rR2enqaQ", "expires_in": 60 }
3Redirect browser with only client_id + request_uri
GET /authorize?client_id=0oa1abc&request_uri=urn:ietf:params:oauth:request_uri:bwc4JD...
No sensitive parameters in the browser URL. No parameter tampering possible. URL length limits no longer a concern.
PAR is mandatory in FAPI 2.0 and strongly recommended for any deployment handling sensitive authorization parameters.

DPoP — Demonstrating Proof-of-Possession (RFC 9449)

What it does

Binds access tokens to a client’s cryptographic key pair. A stolen DPoP token is useless without the private key.

The hotel keycard analogy: Bearer tokens are like hotel keycards — anyone who finds one can open the door. DPoP is a keycard + fingerprint scanner — only the registered guest can use it.

How it works

  1. Client generates an ephemeral EC/RSA key pair
  2. On each token/API request, client creates a DPoP proof JWT signed with its private key
  3. The proof JWT includes: htm (HTTP method), htu (target URL), iat (timestamp), jti (unique ID)
  4. AS issues token with cnf.jkt claim = SHA-256 thumbprint of client’s public key
  5. Resource server verifies: proof signature, cnf.jkt matches proof key, htm/htu match request
// DPoP Proof JWT // Header: { "typ": "dpop+jwt", "alg": "ES256", "jwk": { /* public key */ } } // Payload: { "jti": "unique-id-per-request", "htm": "POST", "htu": "https://auth.example.com/token", "iat": 1712345678, "nonce": "server-provided-nonce" // Required by Okta } // Token response includes: // { "token_type": "DPoP", "access_token": "...", ... } // Access token JWT contains: { "cnf": { "jkt": "SHA256-thumbprint-of-public-key" } }

Rich Authorization Requests — RAR (RFC 9396)

What it does

Replaces flat scope strings with structured JSON authorization_details for fine-grained permissions.

// Instead of: scope=payment_initiate // Use structured authorization_details: authorization_details=[{ "type": "payment_initiation", "locations": ["https://api.bank.example.com/payments"], "actions": ["initiate"], "instructedAmount": { "currency": "EUR", "amount": "123.50" }, "creditorName": "ACME Corp" }]

RAR travels in: authorization requests (via PAR), token responses, and introspection responses. Used extensively in Open Banking (PSD2) and healthcare (SMART on FHIR).

MCP + OAuth: The Full Authorization Architecture

As of the 2025-06-18 spec revision, MCP servers are no longer Authorization Servers. They are OAuth 2.1 Resource Servers only. Understanding this separation is the foundation of every MCP security conversation.

The Three-Party Architecture

OAuth Role
MCP Server

= Resource Server (RS)

Hosts tools, validates incoming Bearer tokens, enforces scope-based access, and publishes its own protected resource metadata.

Validates: signature, expiry, audience (aud MUST match MCP URL), scopes

OAuth Role
MCP Client

= OAuth Client

The AI tool that connects to the MCP server. Examples: Claude Desktop, Cursor, VS Code Copilot, ChatGPT Plugins.

Responsible for: PKCE generation, AS discovery, client registration, token acquisition

OAuth Role
Authorization Server

= Separate Entity (AS)

Issues access tokens. Completely separate from the MCP server. Examples: Auth0, Okta Custom AS, Keycloak, or any RFC 8414-compliant AS.

Endpoint: /.well-known/oauth-authorization-server or OIDC discovery

The Critical Separation (post-2025-06-18): The MCP server no longer doubles as an Authorization Server. It delegates all token issuance to an external AS. This means the MCP server only needs to validate tokens — it never issues them. This is a clean, auditable separation of concerns that mirrors standard OAuth 2.1 best practices.

Protected Resource Metadata (RFC 9728)

RFC 9728 is mandatory as of the 2025-06-18 spec. The MCP server MUST publish a metadata document at a well-known URL so clients can discover which Authorization Servers are trusted and what scopes exist.

GET https://mcp-server.example.com/.well-known/oauth-protected-resource
{
  "resource": "https://mcp-server.example.com",
  "authorization_servers": [
    "https://auth.example.com"
  ],
  "scopes_supported": [
    "mcp:tools:read",
    "mcp:tools:execute",
    "mcp:resources:read"
  ],
  "bearer_methods_supported": ["header"],
  "resource_documentation": "https://mcp-server.example.com/docs"
}

Complete 15-Step Authorization Flow

Click any step to expand the detail. This is the full flow as specified in the 2025-11-25 draft spec — the one you should reference for current client implementations.

1
MCP Client sends unauthenticated request → MCP Server
▼

The client attempts to call a tool or list resources on the MCP server without any Authorization header. This is the initial probe — the client doesn't know yet what authorization is required.

POST /mcp HTTP/1.1
Host: mcp-server.example.com
Content-Type: application/json

{"jsonrpc": "2.0", "method": "tools/list", "id": 1}

MCP Server returns 401 + WWW-Authenticate ← resource_metadata URL
▼

The server responds with a standard 401 and includes a WWW-Authenticate header pointing to the Protected Resource Metadata URL (RFC 9728). This is how the client discovers where to start the auth flow.

HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer realm="mcp-server",
  resource_metadata="https://mcp-server.example.com/.well-known/oauth-protected-resource"
3
Client fetches Protected Resource Metadata RFC 9728 — mandatory
▼

The client GETs the PRM document from the URL in the WWW-Authenticate header. This is an unauthenticated, publicly accessible endpoint on the MCP server.

GET https://mcp-server.example.com/.well-known/oauth-protected-resource
4
PRM contains authorization_servers + scopes_supported Client learns where the AS lives
▼

The PRM response tells the client: (a) which Authorization Server(s) to use, and (b) what scopes the resource server accepts. The client will use the first AS in the authorization_servers array unless it has a preference.

Key field: authorization_servers — an array of AS base URLs. The client will next fetch metadata for the chosen AS using RFC 8414 or OIDC Discovery.

5
Client fetches AS Metadata RFC 8414 or OIDC Discovery
▼

The client retrieves the Authorization Server's metadata to learn the authorization_endpoint, token_endpoint, registration_endpoint, and other capabilities. Two possible discovery URLs:

// RFC 8414 (OAuth AS Metadata)
GET https://auth.example.com/.well-known/oauth-authorization-server

// OIDC Discovery (if the AS is an OIDC provider)
GET https://auth.example.com/.well-known/openid-configuration

Okta uses the OIDC Discovery path. For a custom AS: https://{domain}/oauth2/{authServerId}/.well-known/openid-configuration

6
Client registers (CIMD > DCR > pre-registered) Priority order per 2025-11-25 draft
▼

The client needs a client_id to proceed. Priority order per the 2025-11-25 draft:

  1. CIMD (Client Identity Metadata Document) — SHOULD. Client publishes its own metadata at a well-known URL; AS fetches it. Highest trust, cryptographically verifiable client identity.
  2. DCR (Dynamic Client Registration, RFC 7591) — MAY. Client POSTs a registration request. Requires AS to support open or token-gated registration.
  3. Pre-registered — Fallback. client_id configured in advance. Traditional model; doesn't scale across many MCP servers.
Okta Note: Okta's DCR endpoint (POST /oauth2/v1/clients) requires an API token — it is NOT open registration. This differs from Auth0 which supports open DCR. For MCP clients targeting Okta, CIMD (once supported) or pre-registration is the practical path today.
7
Generate PKCE code_verifier + code_challenge S256 method mandatory as of 2025-11-25
▼

The client generates PKCE parameters before initiating the authorization request. S256 (SHA-256) is mandatory — plain method is not permitted in MCP OAuth as of the 2025-11-25 draft.

// Pseudocode — client-side generation
code_verifier  = base64url(random_bytes(32))   // 43–128 chars
code_challenge = base64url(SHA256(code_verifier))
code_challenge_method = "S256"               // REQUIRED, plain prohibited
8
Open browser to /authorize with resource parameter RFC 8707 — resource=MCP_URL mandatory
▼

The client opens the system browser (or a local redirect server) to the authorization endpoint. The resource parameter (RFC 8707) tells the AS to bind the resulting token to the specific MCP server URL. This is how audience (aud) gets set correctly in the issued token.

GET https://auth.example.com/authorize?
  response_type=code
  &client_id=mcp-client-id
  &redirect_uri=http://localhost:8765/callback
  &scope=mcp%3Atools%3Aread+mcp%3Atools%3Aexecute
  &state=RANDOM_STATE_VALUE
  &code_challenge=BASE64URL_SHA256_OF_VERIFIER
  &code_challenge_method=S256
  &resource=https%3A%2F%2Fmcp-server.example.com  ← RFC 8707
9
User authenticates and consents In the browser — human in the loop
▼

The user logs in to the Authorization Server and is shown a consent screen listing the requested scopes. This is the human-authorization gate. The user sees exactly what access they're granting to the MCP client. After consent, the AS generates an authorization code and redirects.

10
Redirect back with authorization code → Client's local redirect_uri
▼

The AS redirects to the client's redirect_uri with a short-lived one-time authorization code. The client also validates the state parameter to prevent CSRF attacks.

GET http://localhost:8765/callback?
  code=AUTHORIZATION_CODE
  &state=RANDOM_STATE_VALUE  ← client validates this
11
POST /token with code + code_verifier + resource PKCE verification + audience binding
▼

The client exchanges the authorization code for tokens. Crucially, both the code_verifier (PKCE) and the resource parameter (RFC 8707) must be included. The AS verifies the PKCE challenge and uses the resource parameter to set the token's aud claim.

POST https://auth.example.com/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code
&code=AUTHORIZATION_CODE
&redirect_uri=http://localhost:8765/callback
&client_id=mcp-client-id
&code_verifier=ORIGINAL_CODE_VERIFIER      ← PKCE verification
&resource=https://mcp-server.example.com   ← RFC 8707 audience binding
12
Receive access_token with aud=MCP_URL Audience-bound JWT
▼

The AS returns an access token (JWT) whose aud claim is set to the MCP server URL. This audience binding means the token can ONLY be used at this specific MCP server — it will be rejected anywhere else.

// Decoded JWT payload
{
  "iss": "https://auth.example.com",
  "sub": "user@example.com",
  "aud": "https://mcp-server.example.com",  ← MUST match MCP URL
  "exp": 1708387200,
  "scope": "mcp:tools:read mcp:tools:execute"
}
13
Send MCP request with Bearer token Authorization: Bearer {token}
▼

The client retries the original MCP request, this time including the access token in the Authorization header.

POST /mcp HTTP/1.1
Host: mcp-server.example.com
Authorization: Bearer eyJhbGci...
Content-Type: application/json

{"jsonrpc": "2.0", "method": "tools/list", "id": 1}
14
MCP server validates token Signature, expiry, audience, scopes
▼

The MCP server performs full JWT validation. All four checks must pass:

  1. Signature — verified against the AS's public keys from its JWKS endpoint
  2. Expiry (exp) — token must not be expired
  3. Audience (aud) — MUST exactly match the MCP server's own URL. This is non-negotiable per spec.
  4. Scopes — token must include the scope required for the requested tool/resource
Audience validation is mandatory. If the MCP server skips audience checking, an attacker can reuse a token issued for a different resource. The confused deputy attack becomes trivial.
15
MCP response returned Authorized tool execution
▼

Validation passed. The MCP server executes the requested method and returns the JSON-RPC response. Subsequent requests within the token's lifetime reuse the same token — no re-authentication required until expiry.

{
  "jsonrpc": "2.0",
  "result": {
    "tools": [
      {"name": "read_file", "description": "..."},
      {"name": "search_code", "description": "..."}
    ]
  },
  "id": 1
}

Security Vulnerabilities & Threat Vectors

CVE-2025-6514 — Critical RCE in mcp-remote (CVSS 9.6)

The mcp-remote bridge package used to connect local MCP clients to remote servers had an unsanitized authorization URL injection vulnerability. A malicious MCP server could return an authorization_endpoint URL containing shell metacharacters or JavaScript payloads. Because mcp-remote opened auth URLs without sanitization, an attacker could achieve remote code execution on the victim's machine simply by serving a malicious Protected Resource Metadata document. Lesson: always validate and sanitize any URL received from a remote MCP server before passing to system APIs.

Token Passthrough Prohibition

An MCP server MUST NOT forward an incoming access token directly to an upstream API. If it does, it becomes a confused deputy — acting as the user but with the user's full OAuth permissions at the upstream service, not just the scopes needed for the MCP action.

Pattern: Proxy/Bridge MCP servers that call upstream APIs must acquire their own token.

Audience Validation Failure

If an MCP server skips aud claim validation, an attacker who obtains a token issued for a different resource (e.g., a calendar API) can replay it against the MCP server. Resource-binding via RFC 8707 is the preventive control — tokens are audience-locked at issuance.

Always validate: aud === this server's URL.

Prompt Injection → Tool Abuse

An LLM processing untrusted content (web pages, documents, user inputs) can be instructed via injected text to call MCP tools the user never authorized. Even with proper OAuth, the tool call has a valid token — the authorization check passes, but the intent was malicious.

Mitigation: CIBA-style human approval for high-risk tool calls. Scope minimization.

Open Redirect via Authorization URL

A malicious MCP server can return an authorization_servers value pointing to an attacker-controlled AS. The client will send the user there for login, capturing credentials. The proposed URL elicitation extension addresses this with client-side URL validation before navigation.

Status: addressed in roadmap spec; not yet in clients.

Third-Party Authorization Patterns

Three distinct deployment patterns exist for integrating an external Authorization Server with an MCP server. Know which one your customer is building.

Pattern 1
Self-Contained

The MCP server vendor bundles an Authorization Server. The MCP server's domain is both the resource server and hosts the AS.

Pros: Simple deployment, no external dependency

Cons: No enterprise IdP integration, users need a new account, no SSO

Example: A small SaaS MCP server shipping with its own built-in auth

Pattern 2 — Recommended
External AS (Auth0 / Okta)

The MCP server is a pure Resource Server. It trusts tokens issued by an external enterprise AS. PRM authorization_servers points to the enterprise Okta or Auth0 tenant.

Pros: Enterprise SSO, existing IdP policies apply, centralized token management, Okta access policies and inline hooks

Cons: Requires AS configuration; customers must provision Okta app

This is the Okta sales motion target architecture.

Pattern 3
Proxy / Bridge

The MCP server is both a Resource Server (to the MCP client) and an OAuth Client (to an upstream API). It receives a user-bound token from Okta, then acquires its own scoped token to call the upstream API on the user's behalf.

Pros: Scope isolation between MCP layer and upstream API layer

Cons: Complex; must never passthrough tokens directly — must always re-acquire

Example: GitHub Copilot MCP connecting to a corporate API

Okta-Specific Features for Agent Architectures

Token Inline Hooks — Synchronous token customization via external HTTP endpoint ▼

During token issuance, Okta can call an external HTTP endpoint synchronously. Your endpoint receives the token's pending claims and can add, modify, or remove claims before the token is issued. Critical characteristics:

  • Timeout: 3 seconds — your endpoint must respond in under 3s
  • Fail-open by default — if your endpoint times out or returns an error, Okta issues the token WITHOUT the hook modifications. This is a security consideration: a flaky hook won't break auth, but also won't add required claims.
  • Use cases: Add dynamic scopes based on user attributes; inject external authorization decisions (e.g., OPA policy); add agent-specific claims like agent_tier or delegation_level.
// Inline Hook response — add a custom claim
{
  "commands": [{
    "type": "com.okta.access.patch",
    "value": [{
      "op": "add",
      "path": "/claims/agent_delegation_level",
      "value": "restricted"
    }]
  }]
}
Fail-open security note: If your inline hook enforces a security control (e.g., "only issue token if external ABAC system approves"), fail-open means your security control silently fails. Consider using Okta access policies as the primary gate — hooks should augment, not be the sole security mechanism.
Access Policies and Rules — Allowlist-based authorization model ▼

Every Custom AS has access policies. A policy is a set of rules that determine whether to issue a token, for what grant types, with what token lifetimes and scopes. The model is an allowlist — no matching rule means no token.

Rule Conditions (all must match for the rule to fire):

  • Client: All clients, or specific client_ids
  • User: Any user, no user (machine-to-machine), or specific groups
  • Grant type: Auth code, client credentials, device, implicit (deprecated), token exchange
  • Scopes: Which scopes this rule allows to be issued

Token Lifetime Settings (per rule):

  • Access token lifetime: 5 minutes to 24 hours. Default 1 hour. Recommend 5–15 minutes for agent workloads.
  • Refresh token lifetime: Up to indefinite (but should be bounded for agents). Configure rotation behavior.
  • Refresh token rotation: Rotate on use (new RT issued, old RT invalidated after reuse window). Reuse window default: 30 seconds.
For Client Credentials (machine-to-machine): The rule MUST have "User is: Not applicable." A rule that allows any user will NOT match a Client Credentials grant. This is the most common misconfiguration when setting up agent service accounts.
Encrypted Access Tokens (JWE) — Opaque to third parties ▼

Okta supports issuing access tokens as JWE (JSON Web Encryption) instead of plain JWT. The token is opaque to anyone without the decryption key — including the MCP client holding it. The resource server decrypts using its private key registered with Okta.

Use cases for JWE in agent architectures:

  • Prevent agents from reading their own token claims (least-knowledge principle)
  • Hide sensitive claims (user attributes, internal role data) from the client
  • Compliance requirements where token contents must not be logged by intermediaries
Tradeoff: JWE requires the resource server to call Okta's JWKS and perform decryption on every token validation — you lose the "validate locally without network call" benefit of plain JWT. For high-throughput MCP servers, benchmark the decryption overhead.
DPoP (Demonstrating Proof of Possession) — Token binding to client key pair ▼

DPoP binds an access token to the client's private key. Even if the token is stolen, it's useless without the corresponding private key. Okta supports the full RFC 9449 DPoP flow including nonce challenges.

Full Okta DPoP Token Request Flow:

  1. Client generates an asymmetric key pair (EC P-256 or RSA)
  2. Client POSTs to /token with a DPoP header containing a signed proof JWT (htu = token endpoint URL, htm = "POST", iat = now)
  3. If Okta requires a nonce, it returns 401 with DPoP-Nonce header (nonce valid for 24 hours)
  4. Client retries with nonce claim added to the DPoP proof JWT
  5. Okta issues access token with cnf.jkt claim = SHA-256 thumbprint of the client's public key
  6. On each resource request, client sends a new DPoP proof JWT bound to that specific request (htu = resource URL, htm = HTTP method, ath = hash of access token)
  7. Resource server verifies: proof JWT signature, cnf.jkt matches proof key thumbprint, nonce freshness
// DPoP proof JWT header + payload (unsigned example)
{"typ":"dpop+jwt", "alg":"ES256", "jwk":{/* public key */}}
{
  "jti": "UNIQUE_ID_PER_REQUEST",
  "htm": "POST",
  "htu": "https://mcp-server.example.com/mcp",
  "iat": 1708387200,
  "nonce": "OKTA_PROVIDED_NONCE",
  "ath": "BASE64URL_SHA256_OF_ACCESS_TOKEN"
}
Nonce validity: Okta DPoP nonces are valid for 24 hours. Clients should cache the nonce and reuse it across requests within that window. Refresh on 401 with new DPoP-Nonce header.

RFC 8693 Token Exchange on Okta — Agent delegation via token swap ▼

Token Exchange allows a headless agent to obtain a scoped token representing "agent acting on behalf of user" without requiring a browser redirect. The agent presents the user's token plus its own credentials to Okta's token endpoint.

POST https://{domain}/oauth2/{id}/v1/token
Content-Type: application/x-www-form-urlencoded
Authorization: Basic {base64(client_id:client_secret)}

grant_type=urn:ietf:params:oauth:grant-type:token-exchange
&subject_token=USER_ACCESS_TOKEN_OR_ID_TOKEN
&subject_token_type=urn:ietf:params:oauth:token-type:access_token
&scope=read:documents write:calendar
&resource=https://mcp-server.example.com    ← RFC 8707 binding

The resulting token contains:

  • sub = the user's subject (NOT the agent's)
  • act.sub = the agent's client_id (the actor)
  • scope = only the scopes requested (narrower than user's full scope)
  • aud = the MCP server URL (if resource param was included)
The act claim is the audit trail. Every downstream API call made with this token is traceable to both the user (sub) and the agent (act.sub). This is what enables "user Alice authorized agent CRM-Assistant to read documents" in your SIEM.
