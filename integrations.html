Platform Integration Patterns

How Okta and Auth0 integrate with every major AI agent platform — securing agents regardless of where they're built or deployed.

Integration Overview

Every major AI agent platform needs an identity layer. Okta and Auth0 serve as the neutral identity plane that integrates with all of them — so customers aren't locked into any single vendor's auth model, and security teams govern all agent identities from one place.

The Core Message
Okta and Auth0 are platform-agnostic. Whether a customer builds on Salesforce, Microsoft, AWS, Google, or ServiceNow — or all five at once — there is a documented, production-ready integration pattern to secure agent identities from a single control plane.

Salesforce Agentforce + Okta

What is Agentforce?

Salesforce's enterprise AI agent platform — as of 2026, the most widely deployed AI agent runtime in large enterprises. Agentforce agents automate sales, service, marketing, and commerce workflows. They act with Salesforce data, but increasingly reach outside Salesforce to HR systems, ERPs, and internal APIs. This is where Okta governance is essential.

Integration Architecture

OAuth Governance

Agentforce agents authenticate via OAuth 2.0. Okta governs client credentials for each agent — controlling which scopes, resources, and APIs the agent can access, with full token lifecycle management.

ISPM Agent Discovery

Okta ISPM discovers shadow Agentforce agents deployed without IT approval. The Agentforce connector surfaces unauthorized agent deployments alongside your managed agent inventory.

OIG Access Certification

Okta Identity Governance runs access certification campaigns on Agentforce agent permissions. Quarterly reviews ensure agents only retain the access they still need — with full audit trail.

XAA Cross-Platform Delegation

Okta's Cross-App Authentication (XAA) enables Agentforce agents to access non-Salesforce resources — HR systems, ERPs, custom APIs — with governed, time-limited token delegation without credential exposure.

Configuration Steps

  • 1
    Register the Agentforce app in Okta Admin Console Create an OAuth 2.0 application representing the Agentforce org. Use the client credentials grant type for machine-to-machine authentication. Assign to the appropriate group or authorization server.
  • 2
    Configure OAuth scopes for agent capabilities Define granular scopes that map to agent capabilities: agent:read:crm, agent:write:cases, agent:call:erp. Principle of least privilege applies to agents the same as to humans.
  • 3
    Enable ISPM Agentforce connector for agent discovery Connect the Agentforce org to Okta ISPM. The connector continuously scans for agent deployments, surfacing both managed and shadow agents in the ISPM inventory dashboard.
  • 4
    Set up OIG certification campaign for quarterly reviews Configure an access certification campaign targeting Agentforce agent permissions. Assign campaign owners (typically security team or Salesforce admins). Set quarterly cadence with automatic remediation for missed reviews.
SE Positioning — Agentforce Deals
"Salesforce builds the agent. Okta governs ALL the agent's connections — inside and outside Salesforce."

Agentforce gives you agent intelligence. But the moment that agent needs to touch HR data, call an ERP, or access a custom internal API, Salesforce can't govern that. Okta can. This is not an either/or — it's additive to every Agentforce deployment.

Discovery Questions for Agentforce Deals

  1. How many Agentforce agents are currently deployed in your org — and do you have an accurate inventory of all of them? (ISPM angle)
  2. When an Agentforce agent calls your HR system or ERP, what credential does it use? (XAA / credential governance angle)
  3. Who reviews and certifies what Agentforce agents can access? (OIG angle)
  4. Have you had a situation where a department deployed an Agentforce agent without IT knowing? (ISPM shadow agent angle)
  5. What's your process for revoking an Agentforce agent's access when a project ends? (Lifecycle management angle)

Microsoft Copilot Studio + Okta

What is Copilot Studio?

Microsoft's low-code platform for building AI agents (formerly Power Virtual Agents). Copilot Studio supports plugging in ANY OAuth 2.0 identity provider — it is not locked to Entra ID. This is the key insight for Okta positioning: even in Microsoft-heavy environments, Okta secures all the non-Azure connections.

The Key Insight
Copilot Studio supports external OAuth 2.0 providers for custom connector authentication. This means Okta can be configured as the IdP for any Copilot Studio agent that needs to call non-Microsoft APIs. Entra handles Azure-native auth; Okta handles everything else.

Integration Pattern

Okta as External OAuth2 Provider

Configure Okta as the external OAuth 2.0 provider in Copilot Studio. Agents authenticate users via Okta Universal Login — enabling MFA, adaptive policies, and full Okta session governance for agent interactions.

XAA Token Exchange

When a Copilot agent needs to call non-Microsoft resources, XAA handles the token exchange through Okta — issuing scoped, time-limited tokens for each target API without embedding long-lived credentials in the agent.

Entra + Okta Coexistence

Entra ID continues to handle Azure-native authentication. Okta handles the Salesforce, ServiceNow, internal APIs, and any other non-Microsoft resources the agent needs to reach. No migration required.

Architecture Flow

User ──→ Copilot Studio Agent ──→ Okta OAuth 2.0 ──→ Salesforce / ServiceNow / Internal APIs
                             Entra handles Azure resources in parallel
SE Positioning — Microsoft Objection Handler
Objection: "We're a Microsoft shop — we use Entra for everything."

Response: "Your agents don't only talk to Azure. When that Copilot agent calls Salesforce to pull a customer record, or ServiceNow to open a ticket, or your internal HR API — that's not Azure. That's where Okta fills the gap Entra doesn't cover. Copilot Studio already supports external OAuth providers by design. This is the Microsoft-sanctioned integration pattern."

Discovery Questions for Microsoft Deals

  1. Which non-Microsoft systems will your Copilot agents need to call? (XAA / cross-platform delegation angle)
  2. How do you handle MFA and adaptive authentication for users interacting with Copilot agents? (Okta Universal Login angle)
  3. Does your Entra environment cover your Salesforce, ServiceNow, and legacy apps? (Surface the gap)

Amazon Bedrock AgentCore + Auth0

What is Bedrock AgentCore?

AWS's managed runtime for hosting AI agents — launched in 2025. AgentCore provides built-in identity integration via AgentCore Identity, a Token Vault for third-party credential management, and native CIBA support for human-in-the-loop approvals. Auth0 is the documented reference IdP for AgentCore deployments.

Auth0 as the IdP for AgentCore

OIDC Discovery

User authenticates via Auth0 Universal Login. Access token is validated by AgentCore Identity against Auth0's OIDC discovery endpoint — standard OIDC trust establishment, no custom integration code required.

Dynamic Client Registration (DCR)

AgentCore auto-registers OAuth clients in Auth0 via DCR. Each agent instance gets its own client registration — no manual app provisioning. Auth0's DCR API handles the full client lifecycle.

Token Vault Integration

AgentCore's Token Vault stores and retrieves third-party OAuth tokens (Salesforce, GitHub, etc.) without exposing raw credentials to the agent runtime. Auth0 issues the initial token; AgentCore vaults it.

CIBA for Elevated Actions

When an agent attempts a high-risk action, AgentCore triggers a CIBA request through Auth0. This sends a Guardian push notification to the user — they approve or deny the action from their phone before the agent proceeds.

The Three Test Scenarios

Auth0's reference implementation for Bedrock AgentCore covers three key test scenarios that demonstrate the full identity stack:

  • 1
    Token Vault: Agent retrieves third-party tokens without credential exposure The agent requests a Salesforce access token from the AgentCore Token Vault. The vault exchanges the Auth0-issued token for a Salesforce token using stored OAuth credentials. The agent never sees the raw credentials — only the scoped, time-limited token.
  • 2
    CIBA: Agent triggers Auth0 Guardian push for elevated actions When the agent attempts to delete a production record or transfer funds, it initiates a CIBA flow. Auth0 sends a push notification to the user. The agent is blocked until the user explicitly approves — demonstrating human-in-the-loop for high-stakes automation.
  • 3
    FGA: Access denied when authorization tuple is removed mid-session An Auth0 FGA authorization tuple grants the agent access to a sensitive dataset. During an active session, the tuple is deleted. The next resource request returns a 403 — demonstrating real-time FGA enforcement without token revocation.

Configuration Reference

Enable My Account API

Enable Auth0's My Account API in tenant settings. This is required for AgentCore's user self-service token management flow.

Set azp Custom Claim

Configure a custom claim rule to include azp (authorized party) in access tokens. AgentCore Identity uses this to verify the requesting agent's identity.

Configure Enterprise Connection

Set up Okta as an enterprise connection within Auth0 for organizations that use Okta as their corporate IdP. Users SSO through Okta; agents authenticate against Auth0.

SE Positioning — Beat IAM Identity Center
AWS IAM Identity Center is scoped to AWS-native resources. Auth0 handles cross-cloud scenarios — the same agent that calls AWS services can also call Salesforce, Google Workspace, or an on-prem ERP. Show the multi-cloud token delegation scenario in the demo to make this concrete.

Google Vertex AI + Auth0

What is Vertex AI / ADK?

Google's platform for developing and hosting AI agents. The Agent Development Kit (ADK) is Google's open-source SDK for building multi-agent systems. Vertex AI Agent Engine is the managed hosting environment. Google Agentspace is the enterprise deployment layer for Google Workspace users. Auth0 integrates at each level.

Auth0 Integration Pattern

ADK Built-in OAuth 2.0

ADK has native support for OAuth 2.0 authentication on tools that require auth. Configure Auth0 as the OIDC provider in the ADK tool definition — the framework handles the OAuth flow automatically when the tool is invoked.

Vertex AI Agent Engine Identity

In the hosted Agent Engine environment, Auth0 provides the OIDC trust root. Agent Engine validates user-context tokens against Auth0's JWKS endpoint — enabling identity-aware agent execution on Google infrastructure.

Google Agentspace + Auth0 OAuth

For enterprise Agentspace deployments, Auth0 OAuth handles authentication for agents that reach outside Google Workspace — connecting to Salesforce, ServiceNow, internal APIs, and other enterprise systems.

A2A Protocol: Okta/Auth0 + Google Partnership

Agent-to-Agent (A2A) Protocol — May 2025
Okta and Auth0 co-authored the A2A authentication specification alongside Google in May 2025. The A2A protocol defines how AI agents authenticate to each other across platforms and vendors — addressing the trust problem when a Vertex AI agent needs to invoke an Agentforce agent, or vice versa.

This partnership makes Okta/Auth0 the natural identity layer for multi-vendor, multi-platform agent orchestration. Reference: google.github.io/adk-docs/tools-custom/authentication/

ADK Tool Authentication

Configure Auth0 credentials in ADK tool definitions. When the ADK framework invokes a tool requiring auth, it handles the OAuth 2.0 flow — requesting user consent, obtaining tokens, and passing them to the tool.

Cross-Platform Agent Trust

Using A2A, a Vertex AI agent can invoke a tool hosted by an Agentforce agent — with Auth0/Okta providing the trust assertion between the two platforms. This is the future state of enterprise multi-agent orchestration.

SE Positioning — Google Deals
Lead with the A2A partnership. Google and Okta/Auth0 are co-authoring the authentication standard for agent-to-agent communication. Customers building on Vertex AI or ADK who also have Salesforce or Microsoft deployments need a neutral identity layer — and Okta/Auth0 is the co-author of the spec for exactly that scenario.

ServiceNow AI Agents + Okta

The Dual Role

ServiceNow occupies a unique position: it is both a potential competitor in the identity governance space AND a documented integration partner. ServiceNow AI agents have native connectors to Okta APIs — ServiceNow agents call Okta to provision users, manage groups, and automate IT service workflows. Okta governs the credentials those agents use.

ServiceNow AI Agents That Integrate with Okta

Okta Application Management AI Agent

Available in ServiceNow Yokohama Patch 3. This pre-built AI agent automates Okta application provisioning, deprovisioning, and management through natural language ITSM requests.

Okta Group Management AI Agent

Automates Okta group membership changes via ServiceNow workflows. IT staff submit change requests in ServiceNow; the AI agent executes the Okta API calls with governed OAuth credentials.

ITSM Provisioning Workflows

Broader ServiceNow ITSM workflows that invoke Okta APIs for joiner/mover/leaver scenarios — onboarding, role changes, offboarding. Okta is the execution engine; ServiceNow is the orchestration layer.

Integration Pattern

ITSM Request ──→ ServiceNow AI Agent ──→ Governed OAuth Credentials (Okta) ──→ Okta API (Provisioning / Group Management)

The ServiceNow AI agent holds OAuth 2.0 client credentials issued and governed by Okta. All API calls from ServiceNow to Okta use these scoped credentials — enabling full audit trail, scope enforcement, and revocation from the Okta Admin Console without touching ServiceNow configuration.

SE Positioning — ServiceNow Competitive Scenarios
"ServiceNow automates the workflow. Okta governs the identity. You need both."

ServiceNow IGA is a competitive threat in access request and certification workflows. But ServiceNow's own AI agents use Okta APIs. When you see ServiceNow IGA, lead with the integration story: "Your ServiceNow AI agents already call Okta. The question is whether those connections are governed — or just using static admin credentials no one is auditing."
Competitive Handling
If a customer is evaluating ServiceNow IGA to replace OIG: show that ServiceNow's own Okta AI agents depend on a healthy Okta API integration. Replacing OIG with ServiceNow doesn't eliminate Okta — it makes the Okta-ServiceNow integration more critical, not less.
