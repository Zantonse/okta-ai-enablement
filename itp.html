Identity Threat Protection for AI Agents

How do you detect a rogue agent at runtime? ITP monitors post-authentication behavior, detects session hijacking, and triggers Universal Logout when agents are compromised.

1

The Runtime Detection Problem

Traditional security assumes the perimeter. Agents break that assumption entirely — they operate inside the application layer, carry long-lived credentials, and are designed to behave autonomously. By the time a network-level alert fires, the damage is done.

Perimeter vs. Application Layer

Traditional security monitors network perimeter — firewalls, IDS, egress filtering. Agents operate at the application layer, authenticated with valid tokens, making calls indistinguishable from legitimate traffic. Network tools are blind to this threat model.

Token Replay Is Invisible to IP Detection

An agent token stolen from memory or a log file can be replayed from any IP. If an agent normally runs in us-east-1, a replay from eu-west-1 looks like a routine call. IP-based detection misses it entirely.

Authorization Outlives Intent

Agent credentials issued for a specific task frequently persist weeks or months after the task completes. Nobody deprovisioned the access. The agent still holds permissions to systems it no longer needs — a dormant blast radius waiting to be triggered.

Attack Vectors

The four primary vectors: token theft from memory or environment variables, session smuggling via MCP context injection, prompt injection that coerces the agent into requesting elevated scopes, and delegation chain manipulation by inserting a rogue intermediate agent into the act claim.

The Core Gap
Authentication is a point-in-time event. Authorization is a runtime state. Traditional identity tools verify identity at login — ITP is the only product in the Okta portfolio that continuously evaluates whether a session (or agent) should stay authorized after the initial token was issued.
2

For medium-risk sessions, ITP can trigger a Client-Initiated Backchannel Authentication (CIBA) challenge — pushing an out-of-band verification request to the human owner of the agent without interrupting the current session. Third-Party Signal Ingestion

Device risk score and threat intel feeds into ITP. If the host running an agent is marked high-risk by CrowdStrike Falcon, ITP elevates the agent's session risk score automatically — no manual intervention required.

Zscaler ZIA

Internet access risk signals. If an agent is making outbound calls through Zscaler and triggers a threat category match, the risk event propagates into ITP's evaluation engine in real time.

Palo Alto Cortex

Network and endpoint threat signals. Cortex XDR detections can be wired as ITP risk inputs — bridging the EDR signal into the identity risk layer for unified response.

Jamf

macOS/iOS device compliance signals. If an agent runs on a managed device that falls out of compliance (patch missing, disk unencrypted), the ITP risk score for sessions on that device increases.

Universal Logout for Agents

When ITP determines a session or agent is compromised, the response must be total and immediate. Universal Logout does not revoke one token — it terminates every session, token, and credential associated with the identity across every connected system.

How Universal Logout Works for AI Agents

When ITP's behavioral analytics detect anomalous agent activity — for example, an agent accessing 500 Salesforce records in 10 minutes against a behavioral baseline of 10–15 records per day — the risk score crosses the HIGH threshold and automated global token revocation triggers. Universal Logout then revokes tokens across all connected systems simultaneously, not just the system where the anomaly was detected.

This simultaneous cross-system revocation is what distinguishes Universal Logout from a standard token revocation. An attacker who has pivoted to a second system using the same agent token finds both access paths cut at the same moment.

Extended Universal Logout for AI Agents — Phase 2 (GA target FY27)
Today's Universal Logout covers all Okta-connected applications (SAML, OIDC, OAuth). Extended Universal Logout for AI Agents propagates revocation beyond Okta-connected apps — into third-party credential stores, cloud IAM systems, and OPA-managed secrets. This is a Phase 2 capability on the roadmap. The SSF (Shared Signals Framework) event schema is used for cross-vendor interoperability: vendors implement the SSF session_revoked event and receive revocation signals without requiring direct Okta integration.

What Triggers Universal Logout

1
ITP Risk Threshold Exceeded Automated: risk score crosses the configured HIGH threshold, policy fires Universal Logout without waiting for human approval.
2
Admin Manual Action Security analyst uses the Okta Admin Console or API to immediately terminate all sessions for a specific agent client or service account.
3
ISPM Detection Event Identity Security Posture Management identifies an orphaned or shadow agent. The ISPM alert triggers a Workflow that initiates Universal Logout for the rogue entity.
4
OIG Deprovisioning Okta Identity Governance completes an access review that determines an agent's access should be revoked. OIG triggers the lifecycle event, which propagates through Universal Logout.

How It Works

Universal Logout works by revoking the global session at the Okta layer — which cascades token invalidation to every connected application via backchannel logout, token introspection failures, and JWKS key rotation signals. For agent tokens specifically, the access token's jti (JWT ID) is added to a revocation list that resource servers check on every request. The next API call from the agent returns 401. There is no grace period.

Forensic Preservation

Audit Logs Survive Logout
Universal Logout terminates all active sessions and tokens, but does not destroy audit logs. Every API call the agent made, every scope it exercised, every delegation event it participated in — all of this is preserved in Okta's System Log and surfaced in the SecurityEvents API for post-incident forensic investigation. The evidence trail is intact before the first step of your incident response playbook runs.

Agent-Specific Detection Patterns

These are the five patterns most likely to surface in a prospect conversation. For each one: what the signal looks like, what ITP does with it, and what you say to the customer.

Critical Token Replay from Unexpected Geography

An agent's access token is exfiltrated (from environment variables, a log file, or memory) and replayed from a different cloud region or country than the agent's normal operating environment.

What ITP Detects
Access token presented from IP geolocating to eu-west-1 when the agent's historical baseline is exclusively us-east-1. No known cloud provider egress IP for the agent's infrastructure.
Action Fires
Risk score elevated to HIGH. Universal Logout triggered immediately. New tokens blocked until investigation clears. Forensic event logged with full request context.
SE Talking Point
"If one of your agent tokens gets stolen out of an ECS task definition environment variable, ITP is the thing that notices the next API call came from a Berlin datacenter and shuts it down before it can exfiltrate your data."
Critical Scope Escalation

An agent that has historically only exercised read scopes suddenly requests or attempts to exercise administrative scopes included in its token but never previously used. Classic indicator of prompt injection or compromised orchestration logic.

What ITP Detects
Scope usage anomaly: admin:write scope exercised for the first time in the agent's behavioral history. New resource types accessed without precedent in the baseline window.
Action Fires
Medium risk: CIBA step-up sent to agent owner for approval. High risk (if combined with other signals): Universal Logout. Request blocked pending re-authorization.
SE Talking Point
"This is exactly what a prompt injection attack looks like at the identity layer — the agent's token has the scope, the attacker convinced the agent to use it. ITP catches the behavioral deviation even when the token was technically valid."
High Velocity Anomaly

An agent making API calls at 100x its established baseline rate — often indicating a runaway loop, compromised orchestration, or an attacker using the agent token for mass data extraction.

What ITP Detects
Request velocity exceeds 3x rolling baseline within a 5-minute window. Agent baseline: 100 calls/minute. Anomaly: 10,000 calls/minute sustained.
Action Fires
Rate limiting applied immediately at the Okta token introspection layer. Risk score elevated. Alert sent to security team. If velocity continues post-throttle: Universal Logout.
SE Talking Point
"A runaway agent or a stolen token being used for bulk data exfiltration looks the same at the network layer — lots of API calls. ITP distinguishes between them by knowing what normal looks like for that specific agent identity."
Medium Off-Hours Activity

An autonomous agent that historically operates during business hours (driven by human-initiated workflows) suddenly becomes active at 3:00 AM during a scheduled maintenance window when no orchestration jobs should be running.

What ITP Detects
Session activity at 03:17 UTC — outside the agent's historical activity window (08:00–22:00 UTC). No correlated orchestration job ID in request headers. No known maintenance task scheduled.
Action Fires
Risk score elevated to MEDIUM. CIBA challenge sent to agent owner (on-call engineer). If no response within timeout window: session suspended pending review.
SE Talking Point
"Scheduled agent jobs don't just spontaneously run at 3 AM. If an agent is active when it shouldn't be, that's either a compromised credential or a runaway process — and both need human attention immediately."
High Delegation Chain Manipulation

The act claim in a delegated token shows an unexpected intermediate agent — an entity that has inserted itself into the delegation chain and is acting as a conduit for privilege escalation.

What ITP Detects
act.sub claim contains agent-id-unknown-7f3a — an entity never seen before in this agent's delegation history. Chain depth exceeds configured maximum.
Action Fires
Request blocked immediately. Risk elevated to HIGH. Alert generated with full delegation chain decoded for security review. Universal Logout applied to all entities in the chain pending investigation.
SE Talking Point
"Multi-agent systems create a new attack surface — if an attacker can insert a rogue agent into the middle of a delegation chain, they can amplify their permissions through legitimate agents. ITP makes the delegation chain visible and auditable."

ITP + ISPM + OIG Integration

No single product solves the agent identity security problem in isolation. The real power is how ISPM, ITP, OIG, and OPA operate as a unified fabric — each handling a distinct layer of the agent lifecycle.

Product Responsibilities

ISPM — Discovery

Identity Security Posture Management finds agents you didn't know existed. Shadow agents, orphaned service accounts, over-provisioned NHIs — ISPM surfaces them through continuous discovery before they become incident fodder.

ITP — Runtime Monitoring

Identity Threat Protection watches agents you do know about and detects when their behavior deviates from baseline. It acts on the signal in real time — step-up, throttle, or terminate.

OIG — Lifecycle Governance

Okta Identity Governance ensures agents have access reviews, defined owners, expiration policies, and audit trails. When ITP detects a threat, OIG closes the governance loop — triggering review and formal deprovisioning.

OPA — Policy Engine

Open Policy Agent enforces fine-grained authorization decisions at the API layer. While ITP detects and responds to behavioral anomalies, OPA enforces the policy that determines what an agent is allowed to do in the first place.

Current Limitations — Honest Assessment

ITP was built for human session risk. The architecture is correct for agents — but the agent-specific tuning, detection models, and policy controls are still being built. Here is an accurate picture of what works today and what is on the roadmap.

What Works for Agents Today
  • Token-based risk signals (geolocation, velocity, new scope usage, unusual time)
  • Third-party signal ingestion via Shared Signals Framework (CrowdStrike, Zscaler, Palo Alto)
  • Universal Logout — immediate cross-system session and token revocation
  • ITP policies configurable for service account client IDs (agent identifiers)
  • Full audit log preservation for post-incident investigation
  • CIBA-based step-up for medium-risk sessions (requires human owner to be reachable)
What Is on the Roadmap (Phase 2)
  • Dedicated agent behavioral baselines: ML models trained specifically on NHI/agent traffic patterns rather than human session patterns
  • Agent-specific risk policies: A separate policy evaluation path for non-human identities with NHI-native risk factors
  • Extended Universal Logout: Cross-system agent token revocation that propagates beyond Okta-connected apps into third-party credential stores and cloud IAM systems
  • Delegation chain anomaly detection: Native analysis of act claim values and chain depth as first-class risk signals
SE Talking Point for This Section
"ITP gives you runtime visibility into agent behavior today — the token-based risk signals, Universal Logout, and third-party integrations are all available now. The agent-specific detection models and dedicated NHI policy engine are coming in Phase 2. What matters for your architecture conversation is that the infrastructure is already in place — you're not waiting for a new product, you're waiting for new detection models layered on top of a production-grade platform."
