The Shadow AI Problem

Employees are building and connecting AI agents faster than security teams can track them. Unapproved platforms, unsanctioned OAuth grants, and agents running on shared admin credentials are now among the most common unmanaged attack surfaces in enterprise environments. Unsanctioned Platforms

Employees connect personal accounts on ChatGPT Teams, Notion AI, Glean, and other unapproved platforms, granting broad OAuth scopes to company data without IT visibility. Each OAuth grant is a persistent credential that outlives the employee's tenure.

Shadow Agent Credentials

Agentic workflows built in n8n, Zapier, and Make.com routinely run under shared service accounts or personal API tokens with admin-level permissions. No rotation policy. No owner. No lifecycle management.

Platform AI Sprawl

Salesforce Agentforce, Microsoft Copilot, AWS Bedrock, and Google Vertex AI are all creating new classes of non-human identity. Each platform registers agents differently, and none of them feed into a unified identity governance view by default.

Why Traditional PAM and IGA Miss This
Privileged Access Management tools were built for human users and server accounts. IGA tools manage the joiner-mover-leaver lifecycle for humans. Neither was designed to discover OAuth app registrations, analyze scopes granted to AI agents, or track service principals created by a Salesforce flow builder. ISPM fills this gap.

SAM Browser Plugin: Shadow OAuth Detection

The SAM (SaaS Access Management) Chrome extension is the primary mechanism for discovering shadow AI tools. Deployed via Chrome managed policy to corporate devices, it operates at the browser layer where OAuth authorization decisions happen — giving ISPM visibility that no API-level integration can match.

SAM Browser Plugin — Detection & Telemetry Flow
💻

Employee Opens AI Tool

Employee navigates to an AI platform (e.g., Claude.ai, Perplexity, Notion AI) and initiates an OAuth authorization flow using their corporate Google or Microsoft identity.

🔍

Plugin Intercepts OAuth Grant

The SAM extension detects the OAuth redirect and captures the client ID, requested scopes, and resource server. It does not block the flow by default — it observes and reports.

📡

Telemetry Sent to ISPM Console

Grant details are transmitted to the ISPM console: app name, client ID, authorized scopes (read, write, admin), the granting user identity, and timestamp. The app is checked against the approved SaaS catalog.

⚠

Risk Analysis & Classification

ISPM evaluates the grant against policy: Is the app approved? Are scopes excessive relative to declared use? Is this user permitted to grant external OAuth access? Unapproved apps surface as Shadow AI findings in the console.

✅

Remediation Options

Security team can revoke the grant via Okta, add the app to the approved catalog with scope restrictions, or escalate to an OIG access certification campaign to make the owner formally attest the access.

What the Plugin Detects

OAuth Grants to AI Tools

Any OAuth authorization to an external application, with full scope inventory. Captures implicit grants and PKCE flows.

Scope Analysis

Flags overly broad scopes: mail.read, files.readwrite.all, calendars.readwrite. Compares against scope baselines for the app category.

Client-Resource Mapping

Maps the OAuth client (the AI tool) to the resource server it was granted access to. Identifies cases where one AI tool holds access to multiple resource servers.

Deployment

Chrome Managed Policy

Deploy the extension via Google Workspace Admin Console or Microsoft Intune. Force-install to all managed Chrome profiles. No user action required.

Privacy Boundary

The plugin only captures OAuth authorization events and application telemetry. It does not read page content, capture keystrokes, or transmit browsing history.

Identity Correlation

Extension telemetry is correlated with the user's Okta identity, giving full attribution: which employee authorized which AI tool with which scopes.

Key SE Talking Point: Passive Discovery
The SAM plugin is passive by default. It does not block employee productivity or prompt for approval during the OAuth flow. This makes it easy to deploy without change management friction — and means you can get a complete picture of shadow AI before deciding on enforcement policy.

NHI Types Discovered

ISPM surfaces Non-Human Identities across six categories, spanning cloud infrastructure, SaaS platforms, and AI-native identity types. Each type carries its own risk profile and requires different governance treatment.

Service Accounts

Directory service accounts (Active Directory, LDAP) used by applications and automated workflows. Often highly privileged, rarely rotated, frequently shared across systems.

AD service accounts LDAP bind accounts Windows service identities

API Keys

Long-lived secrets used to authenticate to APIs. Embedded in code, CI/CD pipelines, and low-code automation tools. Primary target for secret scanning and rotation enforcement.

OpenAI API keys Anthropic keys Internal API tokens

OAuth Apps & Service Principals

Registered OAuth clients and their associated service principals across cloud identity providers. Each app registration may hold access to multiple resource servers with varying scopes.

Entra service principals Salesforce connected apps GCP service accounts AWS IAM roles GitHub OAuth apps

Salesforce AI Agents

Agentforce agents, Einstein Bots, and Connected App integrations registered in Salesforce. Run under named or shared credentials, often with broad CRM data access and no formal owner.

Agentforce agents Einstein Bots Connected Apps

M2M Application Credentials

Client credentials (client_id + client_secret) used in OAuth 2.0 Client Credentials grant flows. Machine-to-machine authentication for backend services, no human in the loop, frequently long-lived.

client_credentials flows service-to-service tokens CI/CD system tokens

Agentic Workflow Identities

Identities created or used by low-code / no-code automation platforms. Often inherit the creating user's permissions or use a shared platform account, bypassing least-privilege entirely.

Zapier automations Make.com scenarios n8n workflows Power Automate flows

ISPM Agent Discovery is not a standalone product — it is the discovery engine for the full Okta Identity Security Fabric. Every shadow agent ISPM surfaces feeds into the broader governance, vaulting, certification, and threat detection pipeline. This is the complete end-to-end flow.

ISPM → Universal Directory

Shadow agents discovered by ISPM can be formally registered as service accounts or OAuth apps in Okta Universal Directory. This promotes them from "unknown" to "governed" status and assigns an owner for lifecycle management.

UD → Okta Privileged Access (OPA)

Once registered in UD, agent credentials (API keys, service account passwords) are vaulted in OPA. OPA enforces just-in-time access, automatic rotation, and full audit logging for every credential checkout.

UD → Okta Identity Governance (OIG)

ISPM findings can trigger OIG access certification campaigns. The assigned owner is asked to formally attest that the agent's access is appropriate and still required. Unattested access is automatically revoked.

ISPM + ITP: Continuous Monitoring

Identity Threat Protection (ITP) uses ISPM's NHI inventory as context for behavioral anomaly detection. If a newly registered agent starts authenticating from unusual locations or at unusual times, ITP fires a risk signal.

Universal Logout

If a registered agent is compromised — detected by ITP, reported by a security tool, or identified manually — Universal Logout can revoke all active tokens and sessions for that NHI across all integrated systems simultaneously.

SIEM / SOAR Integration

All ISPM findings and remediation events are available via the Okta System Log and API. Feed into Splunk, Sentinel, or Chronicle for correlation with broader security events. Trigger SOAR playbooks on critical severity findings.

The Core Value Prop in One Sentence
ISPM turns shadow AI from a blind spot into a governed asset — and the Okta Identity Security Fabric ensures that "governed" means vaulted, certified, monitored, and revocable.

SAM Plugin: Technical Configuration Details

The SAM (SaaS Access Management) plugin is a managed Chrome extension deployed via Google Admin Console. Understanding its architecture and configuration steps is essential for both demo setup and customer conversations about deployment feasibility.

Plugin Identity & Transport

Extension Type

Managed Chrome Extension, enterprise-deployed via Google Admin Console. Cannot be installed by individual users — requires organizational policy push.

Chrome Extension ID

The extension is identified by the following immutable ID in the Chrome Web Store and in Google Admin Console policy:

Extension ID
galipinbbdandeicdicjbalcbpdbljjj

Data Transport

All telemetry is transmitted via mutual TLS (mTLS) to the Okta org endpoint. The device certificate authenticates the browser to the ISPM backend, ensuring only managed devices can submit grant data.

mTLS Endpoint
https://<org>.mtls.okta.com

Data Captured Per OAuth Grant

Field Description
URL Origin of the OAuth grant (the page where the authorization happened)
Type Grant mechanism: Redirect / Direct / Grant
Okta UserId The Okta user identity of the employee who authorized the grant
Resource The target resource app (e.g., Google Drive, Microsoft 365)
Client The requesting client app — the shadow AI tool seeking access
Scopes Full list of permissions granted (e.g., mail.read, drive.readwrite)
Device / Browser Info Okta Verify Device Trust context: device posture, managed status, browser version

Configuration Walkthrough

Deploying the SAM plugin requires five steps spanning Google Admin Console and the Okta Admin Console. Each step must be completed in order.

1
Chrome Certificate Policy (Google Admin Console)
▶

In Google Admin Console, configure the AutoSelectCertificateForUrls policy so that managed Chrome automatically presents the device certificate to the Okta mTLS endpoint. Without this, the mTLS handshake will fail and no telemetry will be transmitted.

Chrome Policy — AutoSelectCertificateForUrls
AutoSelectCertificateForUrls:
{
  "pattern": "https://<org>.mtls.okta.com",
  "filter": {"ISSUER": {"CN":"Chrome Enterprise CA"}}
}

Replace <org> with your Okta subdomain. The ISSUER CN value must match the CA used to issue your managed Chrome device certificates.

2
Upload CA Certificate to Okta
▶

Upload the Chrome Enterprise CA certificate chain to Okta so the ISPM backend can validate the device certificates presented during mTLS. Navigate to:

Okta Admin Console Path
Security > Device Integrations > Certificate authority tab
> Add certificate authority
> Issue certificate to: Secure Access Monitor plugin
> Upload .pem CA certificate chain

The CA must be the same issuer configured in Step 1. Upload the full chain (intermediate + root) as a single PEM file.

3
Deploy Extension via Google Admin Console
▶

Force-install the SAM extension to the target organizational unit in Google Admin Console. Provide the extension ID and the policy JSON that points the extension at the customer's Okta org URL.

Extension ID
galipinbbdandeicdicjbalcbpdbljjj
Extension Policy JSON
{
  "orgUrl": { "Value": "https://<org>.okta.com" }
}

Replace <org> with the customer's Okta subdomain. This tells the extension which Okta tenant to report to. The extension must be set to Force installed (not allowed/blocked) for automatic deployment.

4
User Enrollment
▶

The SAM plugin activates when two conditions are met simultaneously on the same browser session:

  • 1.The user is signed into a managed Chrome profile (their corporate Google Workspace account via Chrome profile sign-in)
  • 2.The user is signed into the Okta End-User Dashboard (active Okta session in the browser)

When both conditions are met, the plugin can correlate browser OAuth events to the employee's Okta identity. No additional user action or Okta Verify enrollment is required for this step.

5
Verify Deployment
▶

After deploying to a test group, verify that telemetry is flowing to ISPM:

Navigate to the ISPM dashboard and check Inventory → NHI & AI agents → Browser OAuth grants. OAuth grant events from enrolled browsers should appear within the initial data window.

7-Day Initial Data Lag
After first deploying the SAM plugin, allow up to 7 days before shadow AI data appears in the ISPM console. This is a known initial sync window. After the initial population, data syncs daily. Do not diagnose a broken deployment based on the first 24–48 hours of no data.

Shadow AI Detection: Operational Workflow

Once the SAM plugin is deployed and data is flowing, the operational workflow for discovering and acting on shadow AI grants follows a consistent path through the ISPM console. This is the step-by-step process for your first triage session.

Console Navigation Path
ISPM Console > Inventory > NHI & AI agents > Browser OAuth grants
1
Step 1

Navigate to Browser OAuth Grants

Open the ISPM Console and follow the path: Inventory → NHI & AI agents → Browser OAuth grants. This view shows all OAuth grants captured by the SAM plugin across enrolled devices.

2
Step 2

Filter by Category: AI

Apply the Category filter and select "AI". ISPM automatically tags OAuth grants to known AI tools. This narrows the view to shadow AI exposure without manual sorting.

3
Step 3

Review the Inventory Rows

Each row in the filtered view shows: Client app (the AI tool) | Resource app (what it can access) | First seen | Last seen. Sort by First seen to find the oldest persistent grants.

4
Step 4

Drill Into a Grant

Click any row to expand the full grant detail. This reveals: the individual users who authorized this grant, the specific OAuth scopes granted per user, and usage history (last authentication timestamp).

5
Step 5

Choose Remediation Action

For each grant, two remediation paths are available directly from the console:

  • Register — Agent is authorized. Clicking Register promotes the grant to a managed Okta NHI, assigning it an owner and entering it into the governance lifecycle.
  • Revoke — Grant is unauthorized. Revokes the OAuth token immediately, removing the AI tool's access to the resource app.
The Register vs. Revoke Decision
The register/revoke decision is the core governance moment. "Register" signals that the business has reviewed the grant and sanctioned the AI tool — it becomes a managed NHI with an owner, rotation policy, and certification schedule. "Revoke" terminates persistent access. Either action removes the finding from the unresolved queue and creates an audit record.

Known Constraints & Deployment Considerations

ISPM Agent Discovery is in Early Access. The following constraints are important to set accurate customer expectations and avoid demo or deployment surprises. Address these proactively in SE conversations.

!
Chrome-Only: No Firefox, Edge, or Safari
▶

The SAM plugin requires managed Google Chrome deployed via Google Admin Console. There is no equivalent extension for Mozilla Firefox, Microsoft Edge, or Apple Safari. Organizations that do not standardize on Chrome for managed devices cannot use the browser-layer shadow AI detection capability.

Customer Objection Handling
If a prospect uses Edge as their standard managed browser: acknowledge the constraint directly. The Crown Jewel platform connectors (Stage 1) still work regardless of browser. Shadow AI detection via the browser layer requires Chrome. This is a known product gap on the EA roadmap.
!
7-Day Initial Data Lag After Plugin Deployment
▶

After deploying the SAM plugin to managed Chrome profiles, up to 7 days may pass before shadow AI grant data appears in the ISPM console. This is an expected initial sync window, not a bug.

After the initial population period, data syncs on a daily cadence. For demo environments, seed the SAM plugin well in advance (at least one week before the customer meeting) to ensure the console has data to show.

!
Reactive Detection, Not Prevention
▶

ISPM discovers AI agents that have already obtained OAuth grants. The SAM plugin observes and reports authorization events; it does not intercept or block them in real time by default.

Preventing the initial OAuth grant from occurring requires separate policy enforcement — for example, an Okta Access Gateway policy, a Conditional Access policy in Entra ID, or a CASB layer. ISPM is the discovery and governance layer, not the preventive control layer.

Positioning Note
Frame this as intentional design: passive discovery avoids change management friction and gives organizations a full picture of their shadow AI posture before deciding on enforcement. "See everything first, then decide what to block" is a safer rollout than deploying block policies against an incomplete inventory.
!

▶

For EA features (Agent Detection, OIN/Custom App SCIM), customers must request EA access through their Okta account team. EA features are not enabled by default on any org. When positioning these features, confirm EA availability before setting demo expectations. Check the Okta EA changelog and the internal #ispm-ea Slack channel for current status.
FY27 Roadmap Framing
Okta for AI Agents Phase 2 extends the governance lifecycle to include formal Access Certifications campaigns for NHIs (same OIG workflow used for human access reviews) and Extended Universal Logout that can terminate agent sessions across downstream systems — not just Okta-issued tokens. This closes the loop between discovery (ISPM) and full lifecycle governance (OIG + Universal Logout).
