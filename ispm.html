The Shadow AI Problem

Employees are building and connecting AI agents faster than security teams can track them. Unapproved platforms, unsanctioned OAuth grants, and agents running on shared admin credentials are now among the most common unmanaged attack surfaces in enterprise environments. ISPM Agent Discovery (EA February 2026) is Okta's answer.

Unsanctioned Platforms

Employees connect personal accounts on ChatGPT Teams, Notion AI, Glean, and other unapproved platforms, granting broad OAuth scopes to company data without IT visibility. Each OAuth grant is a persistent credential that outlives the employee's tenure.

Shadow Agent Credentials

Agentic workflows built in n8n, Zapier, and Make.com routinely run under shared service accounts or personal API tokens with admin-level permissions. No rotation policy. No owner. No lifecycle management.

Platform AI Sprawl

Salesforce Agentforce, Microsoft Copilot, AWS Bedrock, and Google Vertex AI are all creating new classes of non-human identity. Each platform registers agents differently, and none of them feed into a unified identity governance view by default.

Why Traditional PAM and IGA Miss This
Privileged Access Management tools were built for human users and server accounts. IGA tools manage the joiner-mover-leaver lifecycle for humans. Neither was designed to discover OAuth app registrations, analyze scopes granted to AI agents, or track service principals created by a Salesforce flow builder. ISPM fills this gap.

SAM Browser Plugin: Shadow OAuth Detection

The SAM (SaaS Access Management) Chrome extension is the primary mechanism for discovering shadow AI tools. Deployed via Chrome managed policy to corporate devices, it operates at the browser layer where OAuth authorization decisions happen — giving ISPM visibility that no API-level integration can match.

SAM Browser Plugin — Detection & Telemetry Flow
💻

Employee Opens AI Tool

Employee navigates to an AI platform (e.g., Claude.ai, Perplexity, Notion AI) and initiates an OAuth authorization flow using their corporate Google or Microsoft identity.

🔍

Plugin Intercepts OAuth Grant

The SAM extension detects the OAuth redirect and captures the client ID, requested scopes, and resource server. It does not block the flow by default — it observes and reports.

📡

Telemetry Sent to ISPM Console

Grant details are transmitted to the ISPM console: app name, client ID, authorized scopes (read, write, admin), the granting user identity, and timestamp. The app is checked against the approved SaaS catalog.

⚠

Risk Analysis & Classification

ISPM evaluates the grant against policy: Is the app approved? Are scopes excessive relative to declared use? Is this user permitted to grant external OAuth access? Unapproved apps surface as Shadow AI findings in the console.

✅

Remediation Options

Security team can revoke the grant via Okta, add the app to the approved catalog with scope restrictions, or escalate to an OIG access certification campaign to make the owner formally attest the access.

What the Plugin Detects

OAuth Grants to AI Tools

Any OAuth authorization to an external application, with full scope inventory. Captures implicit grants and PKCE flows.

Scope Analysis

Flags overly broad scopes: mail.read, files.readwrite.all, calendars.readwrite. Compares against scope baselines for the app category.

Client-Resource Mapping

Maps the OAuth client (the AI tool) to the resource server it was granted access to. Identifies cases where one AI tool holds access to multiple resource servers.

Deployment

Chrome Managed Policy

Deploy the extension via Google Workspace Admin Console or Microsoft Intune. Force-install to all managed Chrome profiles. No user action required.

Privacy Boundary

The plugin only captures OAuth authorization events and application telemetry. It does not read page content, capture keystrokes, or transmit browsing history.

Identity Correlation

Extension telemetry is correlated with the user's Okta identity, giving full attribution: which employee authorized which AI tool with which scopes.

Key SE Talking Point: Passive Discovery
The SAM plugin is passive by default. It does not block employee productivity or prompt for approval during the OAuth flow. This makes it easy to deploy without change management friction — and means you can get a complete picture of shadow AI before deciding on enforcement policy.

NHI Types Discovered

ISPM surfaces Non-Human Identities across six categories, spanning cloud infrastructure, SaaS platforms, and AI-native identity types. Each type carries its own risk profile and requires different governance treatment.

Service Accounts

Directory service accounts (Active Directory, LDAP) used by applications and automated workflows. Often highly privileged, rarely rotated, frequently shared across systems.

AD service accounts LDAP bind accounts Windows service identities

API Keys

Long-lived secrets used to authenticate to APIs. Embedded in code, CI/CD pipelines, and low-code automation tools. Primary target for secret scanning and rotation enforcement.

OpenAI API keys Anthropic keys Internal API tokens

OAuth Apps & Service Principals

Registered OAuth clients and their associated service principals across cloud identity providers. Each app registration may hold access to multiple resource servers with varying scopes.

Entra service principals Salesforce connected apps GCP service accounts AWS IAM roles GitHub OAuth apps

Salesforce AI Agents

Agentforce agents, Einstein Bots, and Connected App integrations registered in Salesforce. Run under named or shared credentials, often with broad CRM data access and no formal owner.

Agentforce agents Einstein Bots Connected Apps

M2M Application Credentials

Client credentials (client_id + client_secret) used in OAuth 2.0 Client Credentials grant flows. Machine-to-machine authentication for backend services, no human in the loop, frequently long-lived.

client_credentials flows service-to-service tokens CI/CD system tokens

Agentic Workflow Identities

Identities created or used by low-code / no-code automation platforms. Often inherit the creating user's permissions or use a shared platform account, bypassing least-privilege entirely.

Zapier automations Make.com scenarios n8n workflows Power Automate flows
OWASP NHI Top 10 Coverage
ISPM maps all discovered NHIs against the OWASP Non-Human Identities Top 10 framework (2025). Every finding in the detection library references the relevant OWASP NHI category, giving security teams a standardized risk vocabulary aligned to the industry framework.

From Discovery to Governance: The Identity Security Fabric

ISPM Agent Discovery is not a standalone product — it is the discovery engine for the full Okta Identity Security Fabric. Every shadow agent ISPM surfaces feeds into the broader governance, vaulting, certification, and threat detection pipeline. This is the complete end-to-end flow.

ISPM → Universal Directory

Shadow agents discovered by ISPM can be formally registered as service accounts or OAuth apps in Okta Universal Directory. This promotes them from "unknown" to "governed" status and assigns an owner for lifecycle management.

UD → Okta Privileged Access (OPA)

Once registered in UD, agent credentials (API keys, service account passwords) are vaulted in OPA. OPA enforces just-in-time access, automatic rotation, and full audit logging for every credential checkout.

UD → Okta Identity Governance (OIG)

ISPM findings can trigger OIG access certification campaigns. The assigned owner is asked to formally attest that the agent's access is appropriate and still required. Unattested access is automatically revoked.

ISPM + ITP: Continuous Monitoring

Identity Threat Protection (ITP) uses ISPM's NHI inventory as context for behavioral anomaly detection. If a newly registered agent starts authenticating from unusual locations or at unusual times, ITP fires a risk signal.

Universal Logout

If a registered agent is compromised — detected by ITP, reported by a security tool, or identified manually — Universal Logout can revoke all active tokens and sessions for that NHI across all integrated systems simultaneously.

SIEM / SOAR Integration

All ISPM findings and remediation events are available via the Okta System Log and API. Feed into Splunk, Sentinel, or Chronicle for correlation with broader security events. Trigger SOAR playbooks on critical severity findings.

The Core Value Prop in One Sentence
ISPM turns shadow AI from a blind spot into a governed asset — and the Okta Identity Security Fabric ensures that "governed" means vaulted, certified, monitored, and revocable.

SE Demo Script: ISPM Agent Discovery

A step-by-step demo flow through the ISPM console. Estimated run time: 20–25 minutes. Recommended for CISO-level audiences, security operations leads, and identity governance buyers.

Pre-Demo Setup
Ensure your demo org has the ISPM EA feature enabled. Pre-seed the console with: at least one shadow AI finding (via SAM plugin in the demo browser), a service account with 90+ day stale credentials, and one Salesforce Agentforce agent connected app. The SAM plugin should be installed on the demo machine's Chrome profile.
1
Open ISPM Dashboard — Surface the Problem
▶

Navigate to Security → Identity Security Posture Management in the Okta Admin Console. Land on the ISPM dashboard and let the summary stats tell the story.

"The first thing I want you to notice is this number — [X] unknown non-human identities. These are agents, API keys, and OAuth grants that exist in your environment right now but aren't registered anywhere. That's your shadow AI surface area."

Point to the severity breakdown: Critical / High / Medium findings. Highlight that this view didn't exist before; these risks were completely invisible.

"Before ISPM, the only way to find these was a manual audit. And manual audits happen once a year. Attackers don't wait."
2
Show SAM Browser Plugin in Action
▶

Open a new Chrome tab (with the SAM plugin installed). Navigate to an AI tool — Notion AI or Perplexity works well. Show the OAuth authorization screen. Grant access. Switch back to the ISPM console.

"Watch what just happened. I just authorized an AI tool using my corporate Google account. Let's go back to the ISPM console and refresh the Shadow AI tab."

Show the new finding appearing in near-real time. Expand it to show: app name, authorized scopes (e.g., drive.readonly, gmail.readonly), the user who authorized it, and the timestamp.

"That's the browser plugin doing its job. No API integration with the AI tool needed. No firewall inspection. Just passive observation of where OAuth grants go — which is exactly where shadow AI lives."
Objection: "We already block unapproved apps at the network layer."
Network blocks miss apps accessed over personal hotspots, home networks, or VPN split-tunnel configurations. OAuth grants also persist after the network session ends — the grant survives even if you block the app tomorrow. ISPM captures what's already been authorized, not just what's being accessed right now.
3
Drill Into a Critical Finding: Super Admin API Token
▶

Navigate to Findings → Critical. Select the pre-seeded finding: "Super admin account with active long-lived API token." Expand the full finding detail.

Show: the account name (service-admin@company.com), the token creation date (180+ days ago), the last-used timestamp, the scopes/permissions it grants, and the OWASP NHI category (NHI-01).

"This is a real attack vector. A service account with global admin privileges has an API token that's six months old. If that token is in a script somewhere and the script is in a repo, you have a critical exposure. The attacker doesn't need to phish anyone — they just need to find the token."

Click the Remediate button. Show the options: rotate credential (triggers OPA), assign owner, open in UD, or create OIG certification campaign.

"From here I can rotate the credential immediately into OPA — which means it gets vaulted, the old token is invalidated, and every future checkout is audited. Or I can assign an owner and trigger a certification. The key thing is this is actionable, not just an alert."
Objection: "We have PAM for privileged accounts already."
Traditional PAM was built for human privileged accounts — server admins, DBA accounts. It doesn't understand OAuth app registrations, doesn't discover shadow AI tools, and doesn't have the browser-layer visibility to catch grants made from personal devices. ISPM extends PAM to the NHI dimension that PAM was never designed to cover.
4
Show the NHI Inventory: Crown Jewel Platforms
▶

Navigate to Inventory → All NHIs. Apply the filter: Source = "Salesforce." Show the Agentforce agent connected apps. Click one to show its full profile: owner, last active, scopes, authentication method.

"This is your Salesforce AI surface area. Every Agentforce agent, every connected app, every Einstein Bot — all in one place, with risk scores applied. Notice that this agent has no owner assigned. When the person who built it leaves the company, this thing keeps running. Nobody owns it. Nobody knows it exists."

Switch to the AWS view. Show IAM roles associated with Bedrock agents. Highlight any that have AdministratorAccess or overly broad policies.

"Same pattern across every cloud. AWS, GCP, Azure. The agents were built by developers who needed to move fast. Security governance wasn't part of the build process. ISPM is how you retrofit governance without slowing them down."
5
Close the Loop: Register Shadow Agent into UD
▶

Return to the shadow AI finding from Step 2. Click Register in Universal Directory. Show the registration form: assign an owner, tag with cost center, set credential rotation policy, assign to an OIG certification campaign.

"This is the action that changes the risk posture. We took something that was unknown and unmanaged and made it a governed identity. The owner will get an OIG certification request in their queue. The credential will rotate on a 90-day policy. ITP will start baseline monitoring. Universal Logout can now reach it if it's ever compromised."

Briefly show the OIG certification queue with the new item. Show how the owner receives an email asking them to attest access or revoke.

"The process is familiar to your governance team because it's the same access review workflow they use for humans. We're just extending it to machines. That's the unified identity platform advantage — one workflow, one governance model, for humans and agents alike."
Objection: "This looks like a lot of work for the security team."
The work only happens once per NHI — the initial registration. After that, the governance lifecycle (rotation, certification, monitoring) is automated. And consider the alternative: the work is already happening, but informally and inconsistently, via spreadsheets and ad-hoc Jira tickets. ISPM systematizes work that's already being done badly.
6
Incident Response: Universal Logout for a Compromised Agent
▶

Navigate to the NHI detail view for the registered agent. Show the ITP Risk Score (if elevated in the demo, explain that ITP has detected an anomaly — auth from new IP, unusual time). Click Respond → Universal Logout.

"ITP flagged this agent as authenticating from a new AWS region it's never used before at 2am. That could be a legitimate deployment change or a compromised token being used by an attacker. We don't wait to find out. Universal Logout terminates all active sessions and tokens for this identity — across every integrated system — in seconds."

Show the confirmation: tokens revoked in Okta, downstream app sessions terminated, audit trail created, incident created for investigation.

Objection: "We use CrowdStrike / Defender for threat detection. Why do we need this too?"
Endpoint detection tools see what's happening on devices. They don't see OAuth token usage, service account authentication patterns, or API key usage in cloud-native environments. ITP adds the identity signal layer — and more importantly, Universal Logout is the response capability. CrowdStrike can alert; Okta can revoke. They're complementary.
Demo Closing Frame
End by returning to the dashboard. Show the "resolved findings" count increasing. Frame the narrative: "You started this demo with [X] unknown NHIs and [Y] critical findings. In 20 minutes, you've registered a shadow agent, rotated a stale credential, triggered a governance campaign, and tested your incident response workflow. That's the value of ISPM as the discovery layer for your identity security program."
